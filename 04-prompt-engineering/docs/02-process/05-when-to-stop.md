# PE Process 05：何时停止迭代 —— 性价比与上限

> **一句话**：从 60% → 85% 通常容易，85% → 95% 难，95% → 99% 难到不值得做。本篇讲三个停止信号——**指标收敛、ROI 倒挂、能力天花板**——以及"已经到上限"时的工程化绕过方案。

---

## 1. 一个典型迭代曲线

```
通过率
  ▲
100│                            ▒▒▒▒  ← 95% 上限区，再涨极难
   │                       ▒▒▒▒
 90│                  ▒▒▒▒
   │             ▒▒▒▒
 80│        ▒▒▒▒
   │   ▒▒▒▒
 70│▒▒
   │
   └───────────────────────────────▶ 迭代轮次
       v0  v1  v2  v3  v4  v5  v6  v7  v8 ...

观察：
  v0 → v3：每轮 +5-10 个百分点（最容易的"低垂果实"）
  v3 → v6：每轮 +1-3 个百分点（开始变难）
  v6+：每轮 +0-1 个百分点，甚至倒退
```

**理解这条曲线**比技术细节重要。前几轮"看似越改越好"会让你以为可以一直涨到 100%——其实有上限。

---

## 2. 三个"该停"信号

### 信号 1：指标收敛
连续 3 轮迭代，总通过率 ±1% 内波动 → 已到当前 prompt + 模型的能力上限。

```
v5: 88%
v6: 87% (-1)
v7: 89% (+2)
v8: 88% (0)
v9: 88% (0)
→ 停
```

继续调可能改一处涨 1%，但另一处掉 1%——无收益。

### 信号 2：ROI 倒挂
每轮迭代你能算两个数字：

- **成本**：你的工程时间 + 评测 API 调用
- **收益**：通过率提升 × 业务价值

一开始收益远大于成本；到某一点反过来。

```
轮次     通过率   工程时间   API 成本   业务价值
v3       80%      30 min     $0.5       +5%   ← ROI 大
v6       88%      1 h        $0.5       +1%   ← ROI 小
v9       89%      2 h        $1         +0.5% ← ROI 倒挂
```

ROI 倒挂之后再调，是"工程师的执念"不是"业务需要"。

### 信号 3：能力天花板
所有"剩余失败"的样本都属于**模型本身搞不定**的——和 prompt 关系不大。

```
v8 仍失败的 12 条:
  - 8 条：模型不知道某领域知识（要 RAG / 微调，不是 PE）
  - 2 条：本身有歧义连人都判不准
  - 2 条：模型推理能力到上限（数学多步证明）

剩下的不是 prompt 能解决的。
```

---

## 3. 怎么判定"达标"

不是"达 100% 才能停"——而是"达业务可接受值就停"。

| 任务类型 | 经验上线门槛 |
|---------|-------------|
| 内部工具 / 辅助生产 | 70-80% |
| 客户面对的非关键功能 | 85-90% |
| 业务关键路径（如支付、安全） | 95%+ + 人工兜底 |
| 自动化决策（无人审） | 99%+ |

业务方应该参与定义"什么程度可接受"——而不是工程师自己定。

---

## 4. 达到天花板了——还能做什么

如果 prompt + 模型都到上限，业务还要更高，**改架构**：

### 方案 A：换更强模型
```
Haiku 88% → Sonnet 93% → Opus 96%
```

代价：成本 + 延迟。值不值看业务。

### 方案 B：自洽采样（Self-Consistency）
跑 5 次取多数票，对推理任务能涨 3-8%。
代价：5 倍成本。

### 方案 C：拆分成多步
单 prompt 一次性做"分析 + 决策 + 生成" → 拆成 3 个 prompt。每步更专注，整体准确率涨。
代价：延迟 + 复杂度。

### 方案 D：加 RAG
失败的 8 条是"模型不知道公司内部业务规则" → 加检索增强让相关规则进上下文。

### 方案 E：加 Tool Use
失败的几条是"涉及实时数据 / 精确计算" → 给个工具，让模型调而不是猜。

### 方案 F：加人工兜底
高 confidence → 自动；低 confidence → 转人工。改成 hybrid 流程。

### 方案 G：微调
最贵的方案。仅当上面都不行 + 数据足够（≥ 1000 条）+ 业务足够重要时。

**优先级**：F (人工兜底) > C (拆分) > D/E (RAG/Tool) > B (self-consistency) > A (换模型) > G (微调)。

---

## 5. "卡住了"怎么办的诊断流程

迭代连续 3 轮无进展时按这个流程诊断：

```
[卡住]
   ↓
失败的 10 条样本是不是"同一类问题"？
   ├── 是 → 还有方向，看是哪一类
   │         - 模型本身不会 → 方案 A/G
   │         - 涉及外部知识 → 方案 D
   │         - 需要计算/查询 → 方案 E
   │         - 推理太长 → 方案 C
   │         - 信息歧义 → 改 evalset 或拒绝场景
   │
   └── 否 → 真的到上限了
              ↓
         业务接受当前指标吗？
              ├── 接受 → 上线
              └── 不接受 → 方案 F（人工兜底） + 上线
```

---

## 6. "stop"不是终点——是切换阶段

迭代"停"了不等于工作结束。下一步是 **Stage 5 上线 + Stage 6 监控反哺**：

```
[停止迭代 @ 92%]
   ↓
[Stage 5] 版本化 + 灰度上线
   ↓
[Stage 6] 线上发现新失败模式
   ↓
[新一轮 Stage 4] 针对新失败 → v_{n+1}
   ↓
（持续）
```

evalset 在 Stage 6 长大，prompt 迭代变成"持续性工程"，不是"一次性冲刺"。

## 7. 真实案例：知道何时停的好处

| 项目 | 不知道何时停 | 知道何时停 |
|------|--------------|------------|
| 客服分类器 | 调了 3 周到 94%，团队累 | 1 周到 89%，加人工兜底，整体业务效果一样 |
| 邮件生成 | 调一个月到 80%，开始时间损失大 | 2 周到 70% + 提供"重生成"按钮，用户满意 |
| 代码助手 | 死磕单 prompt 一个月 | 拆 sub-agent，3 天到位 |

**知道何时停 = PE 工程师的成熟度指标**。

---

## 8. 一个 checklist：决定是否停

```
□ 连续 3 轮通过率 ±1% 内？
□ 失败样本是不是都属于"模型不会" / "本身有歧义" / "需要外部数据"？
□ 当前指标业务方接受吗？
□ 继续调一轮的 ROI 是否还正？
□ 是否还有 1-2 个"低成本但可能涨的假设"没试？

5 个里 3+ 个是 → 停
```

---

## 9. 反向：什么时候**不该**停

| 信号 | 含义 |
|------|------|
| 失败仍有清晰 pattern | 还有低垂果实 |
| 业务关键路径没达标 | 必须继续 |
| 出现新的失败模式（线上反哺） | 进入新一轮 |
| 模型刚换 / 上下文刚扩 | 上限已变 |
| 团队认知有共识"我们还能再调" | 信任自己 |

---

## 10. 心理建设

工程师常陷入的陷阱：

| 想法 | 真相 |
|------|------|
| "再多调一点就能 100%" | 95% → 99% 比 70% → 90% 难 10 倍 |
| "停在 88% 显得我没本事" | 知道何时停是更难的本事 |
| "用户会不接受 12% 失败" | 看怎么处理失败——人工兜底就行 |
| "完美主义是工程师的美德" | 在 LLM 不确定性面前是负债 |

PE 是工程，不是艺术。**到上限 + 业务接受 = 完工**。

## 11. 实战 demo：迭代曲线可视化

```python
# demos/process/05_iter_curve.py
"""可视化迭代曲线，判断是否收敛"""
import json
from pathlib import Path


def load_history(results_dir: Path) -> list[dict]:
    out = []
    for f in sorted(results_dir.glob("v*.json")):
        d = json.loads(f.read_text())
        d["version"] = f.stem
        out.append(d)
    return out


def is_converged(history: list[dict], window: int = 3, threshold: float = 0.01) -> bool:
    if len(history) < window:
        return False
    recent = [h["total_pass_rate"] for h in history[-window:]]
    return max(recent) - min(recent) < threshold


def report(history: list[dict]) -> None:
    print(f"{'Version':10s} {'Pass Rate':>10s} {'Delta':>8s}")
    prev = None
    for h in history:
        rate = h["total_pass_rate"]
        delta = "" if prev is None else f"{(rate - prev) * 100:+.1f}%"
        print(f"{h['version']:10s} {rate*100:>9.1f}% {delta:>8s}")
        prev = rate
    print()
    if is_converged(history):
        print("⚠️  最近 3 轮波动 < 1%，已收敛。考虑停止迭代。")
    else:
        print("✅  仍有改进空间，可以继续。")


if __name__ == "__main__":
    import sys
    history = load_history(Path(sys.argv[1]))
    report(history)
```

跑：

```bash
python demos/process/05_iter_curve.py results/
```

输出大致：

```
Version    Pass Rate    Delta
v0             65.0%
v1             74.0%    +9.0%
v2             82.0%    +8.0%
v3             87.0%    +5.0%
v4             89.0%    +2.0%
v5             88.0%    -1.0%
v6             89.0%    +1.0%

⚠️  最近 3 轮波动 < 1%，已收敛。
```

## 12. 常见坑

| 坑 | 排查 |
|----|------|
| **追求 100% 通过** | 永远到不了，也不需要 |
| **不和业务定"可接受门槛"** | 自己 over-engineer |
| **模型能力问题当 prompt 问题** | 换模型 / 加 RAG，不是改 prompt |
| **停了之后不监控** | 上线后 evalset 不更新，等于没建 |
| **见好就收，没做版本化** | 后续维护抓瞎 |
