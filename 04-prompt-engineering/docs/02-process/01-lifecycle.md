# PE Process 01：PE 完整生命周期

> **一句话**：一个好 prompt 的诞生是一个**六阶段的工程闭环**——需求澄清 → baseline → evalset → 迭代 → 上线 → 监控反哺。这一篇把整个流程的骨架画出来，后续 5 篇分别展开每一步。

---

## 1. 全景图

```
              [需求 / Spec]
                   │
                   ▼
     ┌─────────────────────────┐
     │ Stage 1: 需求澄清         │  ← 转成 "通过/失败" 的判定标准
     │ (5-30 分钟)              │
     └────────────┬────────────┘
                  │
                  ▼
     ┌─────────────────────────┐
     │ Stage 2: 写 v0 + 跑      │  ← "最朴素能跑" 版本
     │ (1-2 小时)               │
     └────────────┬────────────┘
                  │
                  ▼
     ┌─────────────────────────┐
     │ Stage 3: 建初版 evalset  │  ← 5-50 条
     │ (2-4 小时)               │
     └────────────┬────────────┘
                  │
                  ▼
     ┌─────────────────────────┐
     │ Stage 4: 迭代闭环         │ ◀─┐
     │ 看失败 → 假设 → 改 → 验证 │   │
     │ (重复 5-30 次)            │   │
     └────────────┬────────────┘   │
                  │                 │
                  ▼                 │
     ┌─────────────────────────┐    │
     │ Stage 5: 上线             │    │
     │ Prompt 版本化 + 灰度       │    │
     └────────────┬────────────┘    │
                  │                 │
                  ▼                 │
     ┌─────────────────────────┐    │
     │ Stage 6: 监控反哺         │    │
     │ 线上失败 → 加进 evalset  │────┘
     └─────────────────────────┘
```

## 2. 六个阶段一表速览

| Stage | 阶段名 | 输入 | 输出 | 时长（首轮） |
|-------|--------|------|------|--------------|
| 1 | 需求澄清 | 模糊需求 | 判定标准 + 数据形态 | 30min |
| 2 | Baseline | 任务 + 几个样本 | prompt v0（能跑出"差不多对的"） | 1-2h |
| 3 | 建 evalset | 业务理解 | 30-50 条 JSONL + 评测器 | 2-4h |
| 4 | 迭代闭环 | evalset + v0 | v_n（达标） | 1-7 天 |
| 5 | 上线 | v_n + CI | 生产版本 + 监控 | 半天 |
| 6 | 监控反哺 | 线上数据 | 新评测样本 + 改进任务 | 持续 |

> ⚠️ 这是**首次**做一个新 prompt 的时间投入。以后做类似任务，evalset 可复用，整个流程会快 2-5 倍。

## 3. Stage 1：需求澄清——把模糊需求转成"通过/失败"

PM / 业务方给你的需求通常长这样：

> "帮我用 LLM 把客户反馈分类成 8 个类别"

这个需求里**所有重要信息都没说**：

- 反馈有多长？1 句话还是 5000 字？
- 8 个类别是什么？互斥还是可多标？
- 模糊样本（同时像 A 又像 B）怎么处理？
- 输出格式？只要类别名还是带置信度？
- 不能分类的输出什么？
- 性能预算？每条几毫秒？每天多少条？
- 错误成本？分错的代价是多少？

不澄清这些就开始写 prompt = 给自己挖坑。

### 正确做法：问 7 个标准问题

```
□ 输入规格：长度范围、格式、语言、来源
□ 输出规格：字段、类型、范围
□ 类别 / 标签：完整清单 + 互斥关系 + 兜底
□ 模糊 / 边界：怎么处理"看起来像 A 又像 B" / 空输入 / 异常
□ 错误成本：误判 A 当 B 损失多大？要不要 confidence？
□ 性能预算：QPS、p99 延迟、每月预算
□ 评测方式：人审 vs 规则 vs LLM-judge 各占多少
```

详细在 [02-from-spec-to-v0.md](./02-from-spec-to-v0.md)。

## 4. Stage 2：Baseline——v0 不需要完美

很多人在 v0 阶段就开始堆技法：CoT、Few-shot、XML 标签、Self-critique 一上来全用。

**别**。v0 应该**最朴素**：

```python
# v0 应该是这样:
prompt = """你是客服反馈分类师。请把以下反馈分类成 8 个类别之一:
[bug, feature_request, complaint, praise, question, billing, account, other]

只返回类别名，不要解释。

反馈: {feedback}
"""
```

为什么不一上来用 CoT、Few-shot？

1. **不知道朴素版能做到什么** → 没有基线 → 不知道高级技法值不值得引入
2. **每个技法都是 cost**（更多 token / 更慢 / 更多复杂度）—— 用之前要证明值
3. **最朴素版本本来就够用的概率 ~40%**

跑 v0 看通过率：

- ≥ 85% → 直接用，不要再调
- 60-85% → 进入迭代
- < 60% → 任务定义或模型选错了（不是 prompt 问题）

## 5. Stage 3：建 evalset——和 v0 同步

evalset 不需要等 v0 完美再建。两者**几乎同时**做：

```
[早 9:00] 澄清需求
[早 10:00] 起草 v0 + 心里默想 5-10 个样本
[中午] v0 写完，跑这 5-10 样本看大致行为
[下午] 把这 5-10 加上更多边界，扩到 30-50，存成 evalset
[傍晚] 跑全量 evalset，拿到 v0 的"基线分"
```

evalset 的 30-50 条是怎么来的？

| 来源 | 比例 |
|------|------|
| 线上真实数据采样 | 50% |
| 人工构造的 edge case | 30% |
| 反例 / 攻击样本 | 20% |

详细在 [03-build-evalset.md](./03-build-evalset.md)。

## 6. Stage 4：迭代闭环——核心循环

```
看 evalset 上失败的样本
   ↓
按失败模式分类（这一组都是长度问题、这一组都是漏字段...）
   ↓
针对**最大的失败类**提出一个假设
   "我觉得加一个明确的长度约束会解决这 5 条"
   ↓
改 prompt（一次只改一处）
   ↓
跑同一份 evalset
   ↓
比较 v_n vs v_{n-1}:
   - 总通过率
   - 上一版失败的本次过了几条
   - 上一版通过的本次挂了几条（regression！）
   ↓
判断：
   - 净正 → 接受 → v_{n+1}
   - 净负 → 回滚
   - 互有胜负 → 看哪边的失败更严重，权衡
```

**铁律**：一次只改一处。改了 3 处，效果变好，你不知道是哪一处的功劳；变差，不知道哪一处该回滚。

详细在 [04-iteration-loop.md](./04-iteration-loop.md)。

## 7. Stage 5：上线——版本化 + 灰度

达标的 prompt 不是文本，是个完整的 "release":

```
prompts/
├── feedback-classifier/
│   ├── v3.0.0/
│   │   ├── system.txt
│   │   ├── user_template.txt
│   │   ├── eval_results.json     # 评测分数留底
│   │   ├── CHANGELOG.md          # 这个版本改了什么
│   │   └── eval_set_ref.json     # 用的哪一版 evalset
│   ├── v3.1.0/
│   │   └── ...
│   └── current → v3.0.0          # 软链接指向当前生产
```

上线流程：

1. **灰度**：1% 流量先用新版本 + 同步 trace 到可观测平台
2. **对比**：拉新旧两套 trace，看分布是否漂移
3. **滚动**：1% → 10% → 50% → 100%
4. **回滚**：发现指标恶化，秒级切换 `current` 软链接

## 8. Stage 6：监控反哺——形成闭环

上线不是终点。生产流量会暴露 evalset 没覆盖的失败模式：

```
[线上] 用户反馈：分类器把投诉当成 praise 处理了
   ↓
[运维] 把这条 input 抓出来
   ↓
[PE 工程师]
   1. 复现失败
   2. 加进 evalset 的 regression.jsonl
   3. 看是否是 evalset 没覆盖的类型 → 加到 happy_path
   4. 评估是否要触发新一轮 v_{n+1} 迭代
```

evalset 是一个**会长大**的资产。开始 30 条，半年后可能 300 条。

## 9. 时间预算参考

一个**首次**做的 prompt 类任务（中等复杂度）：

| 工作 | 时间 |
|------|------|
| 需求澄清 + 几轮 PM 沟通 | 0.5-1 天 |
| 数据采样 + 探索 | 0.5 天 |
| 起草 v0 + 跑通最少样本 | 0.5 天 |
| 建 evalset（30-50 条 + 评测器） | 1 天 |
| 迭代 5-10 轮 | 1-2 天 |
| 上线（版本管理 + 灰度 + 监控） | 0.5 天 |
| **合计** | **4-6 天** |

如果是 **复制类似任务**（已有 evalset 框架），可以压到 1-2 天。

如果有人**跟你说 1 天能搞定**——多半是没建 evalset。

## 10. 不同任务类型的流程裁剪

不是所有任务都要走完整 6 阶段：

### 简单分类（10 个类别内）
1-2 阶段可以快速合并；evalset 主要是规则就够。1-2 天。

### 抽取（多字段）
完整流程。重点在 evalset 覆盖各种字段缺失 / 格式异常。3-5 天。

### 生成（写文案 / 邮件）
LLM-as-judge 是核心评测手段；要做"自由度 vs 受控"的平衡。1-2 周。

### Agent / 长链
evalset 是"完整 trajectory" 而非单步 IO；评测维度包括 "task success"、"step-level correctness"。2-4 周。

### 多模态
评测要拉视觉专家或多模态 judge。3 天起。

## 11. 反例：跳过流程的代价

下面这些"我赶紧上线"导致的真实事故：

| 事故 | 跳过的阶段 |
|------|-----------|
| 客服分类器上线后把 30% 投诉分进 praise | Stage 3 没覆盖含反讽的样本 |
| 内容审核 prompt 改完上线，把正常内容拒了 | Stage 4 没回归，只看通过率不看 regression |
| Prompt 改坏了团队没人能回滚 | Stage 5 没版本化 |
| 月底成本超预算 5 倍 | Stage 1 没问性能预算 |

每个事故都对应着流程某一步被偷懒。

## 12. 怎么"启动"这个流程

如果你现在有一个待做的 prompt 任务：

1. **打开一份 Notion / 文档**，标题 `<feature-name> PE Spec`
2. **写下 7 个标准问题的答案**（[问题清单](#3-stage-1需求澄清把模糊需求转成通过失败)）
3. **新建一个 git repo / 子目录**：
   ```
   feature-name/
   ├── spec.md          ← Stage 1 输出
   ├── prompts/
   │   └── v0/
   ├── evalset/
   │   └── samples.jsonl
   ├── eval_runner.py
   └── results/
   ```
4. **进入 Stage 2-3**（往下读对应章节）

## 13. 常见坑

| 坑 | 排查 |
|----|------|
| **跳过 Stage 1 直接写** | 中途反复发现"诶这个需求我没问清"，全部白做 |
| **v0 就上各种高级技法** | 不知道高级技法值不值得引入 |
| **没建 evalset 就调 prompt** | 永远不收敛 |
| **一次改多处 prompt** | 不知道哪一改的功劳 / 锅 |
| **达标了不版本化** | 半年后某次改坏没法回滚 |
| **上线后不监控** | 线上漂移看不见 |
| **evalset 半年不更新** | 评测分高但用户骂 |
