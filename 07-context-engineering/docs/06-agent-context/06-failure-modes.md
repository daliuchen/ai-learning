# 上下文相关的 Agent 失败模式

> **一句话**：Agent 跑着跑着突然变蠢、绕圈、坚持一个错误答案，根因常常不在模型也不在 prompt，而在**上下文被污染、被带偏、装进了矛盾信息**——认识这几类典型失败模式，才能对症诊断和恢复。

---

## 1. 四类核心上下文失败

这是 2025 年起被广泛讨论的「context failures」分类，记住这四个名字能让你定位问题快一倍：

| 失败模式 | 含义 | 一句话识别 |
|----------|------|-----------|
| **Context Poisoning（中毒）** | 错误信息（幻觉、错误工具结果）进了上下文，此后一直被当真，反复误导 | 模型坚持一个错事实，怎么纠正都绕回去 |
| **Context Distraction（分心）** | 大量无关 / 过期内容塞满窗口，把模型从当前目标带偏 | Agent 跑题、被旧内容勾走、忘了在干嘛 |
| **Context Clash（冲突）** | 上下文里同时存在互相矛盾的信息 | 模型在两个说法间反复横跳、自相矛盾 |
| **Context Distraction → 退化** | 窗口过长触发 context rot，整体注意力摊薄 | 步数越多越笨、重复已做过的动作 |

它们常常连锁：一个错误工具结果（poisoning）+ 越积越长的轨迹（distraction）+ 一次纠正引入的新说法（clash），叠起来就是一个彻底跑飞的 Agent。

---

## 2. Context Poisoning：错误进了上下文就阴魂不散

最危险的一类。一旦某个 step 产生了错误信息——模型幻觉、工具返回了错数据、或抓到一个过期网页——这条内容留在上下文里，**后续每一步都把它当既定事实**。

```
step 3: 工具返回 "pydantic 最新版是 1.10"（其实早过时了）
step 4~20: 模型基于 "1.10" 做了一连串决策、写了不兼容的代码
         你纠正它，它道歉，但上下文里 "1.10" 还在，下一步又用回去
```

- **症状**：纠正无效，模型像「记忆固化」一样反复回到错误结论。
- **对策**：不能靠"再说一遍正确答案"压制——必须**把污染源从上下文里清掉**（删掉那条错误工具结果 / 截断到污染发生前），或者**隔离重启**子任务（见 [04-isolation.md](./04-isolation.md)）。这也是工具结果要可定位（带句柄）的原因之一——出了错能精准移除。

---

## 3. Context Distraction：被噪声带偏

窗口里塞了太多无关或过期内容（典型就是累积的工具结果、早期已无关的轨迹），模型的注意力被分散，开始关注不该关注的东西，甚至忘记最初目标。

```
# ❌ 一条上下文里堆了 40 步的全部 grep 结果、网页全文、试错日志
# 模型在第 41 步被第 12 步的某个无关片段勾走，跑去做一件不相干的事

# ✅ 对策：主动裁剪 + state 外置 + 工具结果落盘
#   - 旧的、已用完的工具结果摘要化或移除（见 02 / 05 篇）
#   - 把目标、待办放外部 state，每步只注入"当前焦点"（见 05 篇）
```

- **症状**：跑题、被旧内容勾走、目标漂移。
- **对策**：源头减少累积（[01](./01-accumulation.md)、[02](./02-tool-results.md)），并在窗口变长时压缩历史（[../05-compaction/](../05-compaction/01-why-compact.md)）。

---

## 4. Context Clash：矛盾信息打架

上下文里同时存在互相矛盾的内容——比如检索注入了两份说法不一致的文档，或一次纠正后新旧两个答案都还在窗口里。模型不知道信哪个，于是反复横跳或胡乱融合。

- **症状**：自相矛盾、答案在两个版本间摇摆、把矛盾的两边硬拼在一起。
- **对策**：注入前**去重 / 取舍**（保留更可信、更新的来源，剔除矛盾的旧的）；纠正时不要"叠加"新说法，而要**替换**掉旧的；多源检索结果带时间戳和可信度，让模型有取舍依据。

---

## 5. 工具结果错误累积 与 循环

两个高频的衍生故障：

- **工具结果错误累积**：单次工具错误不致命，但 Agent 不察觉、继续基于错结果调更多工具，错误像滚雪球。对策：工具结果做健全性校验（schema 校验、空结果检测），异常时**显式标注失败而非静默塞回上下文**，必要时回滚到出错前。
- **循环（loop）**：Agent 反复执行同一个无效动作（同样的搜索、同样的失败调用）。根因常是上下文里"上次这么做失败了"的信号太弱、或被噪声淹没。对策：在 state 里记动作历史 + 显式去重（见 [05-state-vs-context.md](./05-state-vs-context.md)），检测到重复动作就强制换策略或终止。

```python
# ✅ 用外部 state 做循环检测（不靠模型自己在长上下文里"想起来"）
def guard_against_loop(state, action_signature: str) -> bool:
    recent = state["action_history"][-5:]
    if recent.count(action_signature) >= 3:     # 同一动作近期重复 3 次
        return True   # 触发：强制换路或终止，并往上下文注入明确提示
    return False
```

## 6. 失败模式 → 症状 → 对策 速查表

| 失败模式 | 典型症状 | 诊断信号 | 对策 |
|----------|----------|----------|------|
| **Context Poisoning** | 坚持错误事实、纠正无效 | 错误结论可追溯到某条工具结果 / 幻觉 | 清除污染源（删 / 截断到污染前）；隔离重启子任务 |
| **Context Distraction** | 跑题、目标漂移、被旧内容勾走 | 窗口过长、塞满无关历史 | 工具结果落盘 + 摘要；裁剪旧内容；目标放 state |
| **Context Clash** | 自相矛盾、答案摇摆 | 上下文含互斥说法 | 注入前去重取舍；纠正用替换非叠加；带来源/时间戳 |
| **Context Rot（窗口过长退化）** | 步数越多越笨、重复动作 | 上下文逼近窗口上限 | 压缩历史（见压缩章）；拆多 Agent 隔离 |
| **工具结果错误累积** | 基于错结果滚雪球 | 某次工具返回异常未被察觉 | 结果校验；显式标失败；回滚 |
| **循环** | 反复同一无效动作 | action 重复 | state 记动作历史 + 去重检测 + 强制换策略/终止 |

---

## 7. 通用恢复三板斧

诊断清楚后，恢复手段无非这三类，按代价从轻到重：

- **清理（trim）**：移除 / 摘要化污染或无关内容，保留有效轨迹。适合污染源明确、损失可控。
- **重置（reset）**：截断到某个干净检查点，或带着"已确认事实"的精简摘要重开一段上下文。适合污染已扩散、清理成本高。
- **隔离（isolate）**：把出问题的子任务丢进一个全新的独立窗口重跑，结果只回传结论（见 [04-isolation.md](./04-isolation.md)）。适合需要彻底切断污染传染链。

```
轻 ──────────────────────────────▶ 重
清理            重置            隔离重跑
(删污染源)   (截到检查点)    (全新窗口重做子任务)
```

经验：**与其事后救火，不如从源头少累积、早隔离、勤外置**——本章前五篇讲的全是预防。这一篇是当预防没做够、事故已发生时的诊断与急救手册。
