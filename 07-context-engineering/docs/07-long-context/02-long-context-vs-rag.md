# CE 07-02：长上下文 vs RAG——还要不要检索

> **一句话**：窗口都 1M 了，能直接把整个知识库塞进去，还要不要 RAG？答案是「看情况，而且多数情况下还要」。长上下文不是 RAG 的替代品，是 RAG 的补充——知识库小且稳就直接塞，大而动态就检索，最强的往往是「RAG 粗筛 + 长上下文精读」的混合方案。

---

## 1. 这个问题为什么会被问出来

RAG（检索增强生成）诞生的一个核心动机，是**早年窗口太小**——4K、8K 根本塞不下一个知识库，只能检索出最相关的几段塞进去。

现在窗口 1M 了，自然有人想：那我直接把 300K 的文档全塞进去，让模型自己读，不就省掉了切块、embedding、向量库、检索这一整套复杂工程了吗？

这个想法**部分成立**，但不能一刀切。下面拆开维度看。

---

## 2. 决策维度逐个看

### 2.1 知识库大小

- 知识库 **< 窗口的一半**（留出输出和 rot 余量）：可以考虑直接塞。
- 知识库 **远超窗口**（几百 MB、几 GB 的企业文档库）：物理上塞不下，**必须 RAG**。这是最硬的约束。

### 2.2 更新频率

- **几乎不变**（一份产品手册、一套法规）：塞进去 + prompt caching，缓存命中后又快又便宜。
- **频繁更新**（每天新增文档、实时数据）：RAG 更新索引很容易，长上下文每次都得重新塞 + 缓存失效，反而麻烦。

### 2.3 成本

每次调用都塞 300K 输入，按量计费会**线性累加**。RAG 只塞检索出的几 K～几十 K，单次成本低一个数量级。

> 长上下文配 prompt caching 能把「重复塞同一份文档」的成本压下来；但「每次塞不同的大文档」省不了。

### 2.4 延迟

长输入的 **prefill 时间长**，首 token 延迟高。RAG 的输入短，prefill 快——但要加上检索那一跳的耗时（通常几十～几百毫秒，远小于塞 300K 的 prefill）。

### 2.5 准确率

- **单针检索（找一个明确答案）**：长上下文大海捞针能做得很好。
- **综合 / 多跳推理**：长上下文会触发 **lost-in-the-middle**，埋中间的信息读不到，准确率掉点。RAG 把最相关内容前置，反而稳。
- RAG 的风险在**检索召回**：检索没召回到的内容，模型根本看不到。

### 2.6 可解释性 / 引用

RAG **天然带溯源**——你知道答案来自哪个 chunk、哪篇文档，可以做引用、做审计。长上下文是「整坨塞进去」，模型说的话来自哪一段很难追溯（除非额外做 citation 机制）。

合规、法律、医疗这类**必须可追溯**的场景，RAG 的可解释性是刚需。

---

## 3. 决策表

把上面拍成一张表，按场景对号入座：

| 维度 | 倾向「直接长上下文」 | 倾向「RAG」 |
|------|---------------------|------------|
| 知识库大小 | 小，能塞进半个窗口内 | 大，远超窗口 |
| 更新频率 | 几乎不变 | 频繁更新 / 实时 |
| 单次成本 | 配缓存、重复用同一份 | 每次内容不同、要省钱 |
| 延迟要求 | 可接受秒级 prefill | 要低延迟 |
| 任务类型 | 单针检索、整体摘要 | 多跳、需要精准定位 |
| 准确率 | 内容短、关键信息能放头尾 | 内容长、怕 lost-in-middle |
| 可解释性 / 引用 | 不强求溯源 | 必须可引用、可审计 |
| 工程复杂度 | 想简单、不想维护向量库 | 已有检索栈、能接受复杂度 |

> 经验法则：**「小而稳」→ 长上下文；「大而动 / 要溯源 / 要省钱」→ RAG；介于中间 → 混合。**

---

## 4. 混合方案：RAG 粗筛 + 长上下文精读

两者不是二选一。最强的工程实践往往是**用长窗口放宽 RAG 的精度要求**：

- 过去窗口 4K，RAG 只能塞 top-3，对检索精度要求极高（漏一个就完）。
- 现在窗口 200K+，RAG 可以**粗筛召回 top-50 甚至 top-100**，把召回率拉满，剩下的「精读 + 筛选」交给长上下文模型自己做。

这就是 **「retrieve wide, read long」**：检索负责「不漏」，长上下文负责「读懂 + 挑出真正相关的」。

```python
# ✅ 混合：RAG 粗筛召回，长上下文精读
import anthropic

client = anthropic.Anthropic()

# 1) RAG 粗筛：召回多一点（top_k 拉大，重召回轻精度）
chunks = vector_store.search(query, top_k=50)   # 召回 50 段，约 60K token

# 2) 长上下文精读：把召回的 chunk 全塞给模型，让它自己判断相关性 + 综合作答
context = "\n\n".join(
    f"[来源 {c.doc_id} #{c.chunk_id}]\n{c.text}" for c in chunks
)
resp = client.messages.create(
    model="claude-opus-4-20250514",
    max_tokens=1500,
    messages=[{
        "role": "user",
        "content": (
            f"以下是检索到的资料，请只依据资料作答，并在每个结论后标注来源编号。\n\n"
            f"{context}\n\n问题：{query}"
        ),
    }],
)
```

注意上面给每段加了 `[来源 doc_id #chunk_id]` 标记——这就**把 RAG 的可溯源性带进了长上下文**，让模型能引用来源，弥补纯长上下文不可解释的短板。

## 5. 几个场景判断

| 场景 | 选型 | 理由 |
|------|------|------|
| 问答一份 50 页产品手册（基本不变） | 长上下文 + 缓存 | 小而稳，缓存命中后又快又便宜 |
| 企业 10 万篇文档客服知识库 | RAG（可粗筛 top-50 给长上下文精读） | 远超窗口，物理上必须检索 |
| 合同审查，结论必须引用条款 | RAG / 混合 | 强可解释、要溯源到具体条款 |
| 分析单个超长代码仓库做重构建议 | 长上下文 | 需要全局理解，跨文件依赖塞进去更好 |
| 实时新闻问答 | RAG | 内容每天变，索引更新比重塞缓存省事 |
| 单篇长论文总结 | 长上下文 | 单文档、要全局综合、不需要检索 |

---

## 6. 常见坑

| 坑 | 说明 |
|----|------|
| 「窗口大了就不用 RAG 了」 | 大知识库物理塞不下；动态数据、可解释性仍需 RAG |
| 把全库塞窗口求省事 | 成本线性涨、prefill 慢、还触发 lost-in-the-middle |
| 混合方案 top_k 还卡得很小 | 长窗口的意义就是放宽召回，召回多一点交给模型精读 |
| 长上下文不做溯源 | 合规场景不可追溯是硬伤，至少给 chunk 加来源标记 |
| 忽略缓存 | 长上下文重复用同一份文档时，不开缓存白白多花钱 |
