# Few-shot 示例：用预算换确定性

> **一句话**：few-shot 示例是上下文里**花预算买稳定输出**的一块，通常 1–5 个就够；它在"格式/风格/边界"上的提质效果显著，但要权衡占用——示例 vs 微调 vs 动态检索示例，三条路按场景选。

---

## 1. few-shot 在上下文里的角色

few-shot 就是在提示里给几组"输入 → 期望输出"的范例，让模型照葫芦画瓢。它占的 token 是**为了换取输出确定性**而主动支付的预算。

| 手段 | 怎么教模型 | 占上下文 |
|------|-----------|---------|
| **zero-shot** | 只给指令 | 不占 |
| **few-shot** | 指令 + 几个示例 | 占（每个示例几十~几百 token） |
| **微调** | 改权重 | 不占上下文 |

示例最擅长传达"**难以用文字说清的隐性要求**"——输出格式、语气风格、边界 case 的处理。你说十句"要简洁专业"，不如给一个简洁专业的范例。

---

## 2. 放 system 还是 user？

| 放法 | 适合 | 说明 |
|------|------|------|
| **放 system** | 示例稳定、全程通用 | 当成"格式规范"的一部分，且**可进缓存** ✅ |
| **放 user**（多条 user/assistant 交替） | 模拟真实多轮、示例随 query 变 | 更贴近对话格式 |

```python
# ✅ 稳定示例放 system，吃满 prompt caching
SYSTEM = """你是情感分类器，输出 positive/negative/neutral 之一。

示例：
输入：这手机续航太顶了 → positive
输入：客服半天不回，差评 → negative
输入：包装一般，东西还行 → neutral"""

# 或用 messages 交替模拟（适合对话/工具调用类示例）
messages = [
    {"role": "user", "content": "这手机续航太顶了"},
    {"role": "assistant", "content": "positive"},
    {"role": "user", "content": "客服半天不回，差评"},
    {"role": "assistant", "content": "negative"},
    {"role": "user", "content": "<待分类的真实输入>"},   # 真问题放最后
]
```

---

## 3. 几个示例够？通常 1–5 个

| 示例数 | 效果 |
|--------|------|
| 0（zero-shot） | 现代强模型在简单任务上常已够用 |
| **1–5** | **性价比最高**：格式/边界基本锁定，预算可控 ✅ |
| 10+ | 边际收益递减，且占预算、可能引入冲突信号、稀释注意力 |

经验：**先 zero-shot 跑，效果不稳再逐个加示例**，而不是一上来堆 10 个。加到"输出稳定"就停。任务越复杂/格式越特殊，需要的示例越多，但很少超过 5 个。

```text
输出格式跑歪了      → 加 1 个标准格式示例
边界 case 处理错了  → 补 1 个该 case 的示例
风格不对           → 加 1 个目标风格示例
```

---

## 4. 三条路线：示例 vs 微调 vs 动态检索示例

| 路线 | 占上下文 | 灵活性 | 适合 |
|------|---------|--------|------|
| **静态 few-shot** | 占（固定几条） | 改提示即可 | 任务稳定、示例少 |
| **微调** | 不占 | 改一次定型 | 海量同类样本、追求极致一致性/低延迟 |
| **动态 few-shot（检索式）** | 占（按 query 召回） | 最高 | 示例库大、不同输入需不同范例 |

**动态 few-shot** 是工程上常用的进阶：维护一个示例库，每来一个 query，**检索最相似的 k 个示例**注入上下文。既不堆满预算，又让示例和当前问题高度相关。

```python
def build_dynamic_fewshot(query, example_store, k=3):
    examples = example_store.search(query, top_k=k)   # 向量检索最相似示例
    shots = "\n".join(f"输入：{e.input} → {e.output}" for e in examples)
    return f"参考示例：\n{shots}\n\n输入：{query} →"
```

> 取舍：静态示例可进缓存（前缀稳定）；动态示例每次变，**牺牲缓存换相关性**。示例库 < 几十条用静态，上百上千条且差异大用动态。

## 5. 示例的多样性与顺序

不是随便挑几条，质量比数量重要：

| 维度 | 要点 |
|------|------|
| **多样性** | 覆盖不同类别/边界，别全是 positive 让模型以为只有一种答案 |
| **代表性** | 选典型而非极端 corner case 当主示例 |
| **平衡** | 分类任务各标签数量大致均衡，避免诱导偏置 |
| **顺序** | 模型对**最后一个示例**更敏感（近因效应），别把最不典型的放末尾；分类任务避免按标签排序造成模式泄漏 |
| **一致性** | 所有示例的格式、风格必须和你想要的输出完全一致——示例就是"事实标准" |

```python
# ❌ 示例标签全是 positive，顺序又按标签聚堆
shots = ["好评1→positive", "好评2→positive", "差评→negative"]

# ✅ 标签均衡 + 打散顺序
shots = ["差评→negative", "好评→positive", "中评→neutral", "好评→positive"]
```

---

## 6. 常见坑

| 坑 | 后果 | 对策 |
|----|------|------|
| 一上来堆 10+ 示例 | 占预算、边际收益低、稀释 | 从 0/1 起，加到稳定就停 |
| 示例格式和期望输出不一致 | 模型学偏 | 示例 = 输出的事实标准 |
| 标签/类别不均衡 | 输出偏置 | 各类均衡 |
| 动态示例还放进 system | 打碎缓存 | 动态的放 user |
| 示例过时（含旧政策/旧格式） | 误导 | 定期审查示例库 |

---

## 7. 小结

- few-shot 是**花预算换输出确定性**，最擅长传达难以言说的格式/风格/边界。
- 稳定示例放 system（可缓存），多轮/动态示例放 user。
- 通常 **1–5 个**够用，从 zero-shot 起，加到稳定就停。
- 示例库大就上**动态 few-shot**（检索最相似的几个），用缓存换相关性。
- 多样性、平衡、顺序、一致性都影响效果——示例就是模型眼里的"标准答案"。
