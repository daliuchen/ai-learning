# CE 08-03：上下文相关故障排查

> **一句话**：模型出问题时，工程师的本能是改 prompt 措辞——但 80% 的生产故障根因不在措辞，而在上下文：无关内容稀释了注意力、不同来源的信息打架、关键指令被埋在长历史中间、检索压根没命中。排查的第一步永远是「dump 出模型实际收到的完整上下文」，而不是盯着模板猜。

---

## 1. 黄金第一步：先 dump 上下文，再谈别的

90% 的上下文故障，肉眼扫一遍渲染后的完整 messages 就能定位。**在改任何代码前，先把模型实际收到的东西打出来。**

```python
import json
# messages 是即将发给 API 的最终序列（不是你的模板）
with open("/tmp/dump_ctx.json", "w") as f:
    json.dump({"messages": messages, "tools": tools}, f, ensure_ascii=False, indent=2)
print(f"system 长度 {len(str(messages[0]))} 字符，共 {len(messages)} 条消息")
```

扫这份 dump 时盯四件事：

1. 关键信息（用户约束、检索到的事实）**在不在**里面？
2. 在的话，**在哪个位置**？（开头/结尾 vs 中间）
3. 有没有**互相矛盾**的内容（旧检索 + 新检索、历史里的旧事实 + 新事实）？
4. 有没有**一大坨无关内容**把信号淹没了？

这四问基本对应下面四类典型故障。

---

## 2. 症状 → 根因映射

### 2.1 答非所问 / 跑题 → context rot 或无关上下文

模型回答和问题对不上，或者抓着上下文里某个无关细节展开。多半是窗口塞了太多噪声，关键信号被稀释——这就是 context rot（详见 [../01-foundations/03-context-rot.md](../01-foundations/03-context-rot.md)）。

**自检**：dump 里有效信息占比多少？如果检索召回了 10 条但只有 1 条相关，那 9 条就是在抢注意力。

### 2.2 前后矛盾 / 自相打架 → context clash

模型一会儿说 A 一会儿说非 A。根因通常是上下文里同时存在冲突信息：第一轮检索说「退货 7 天」，第三轮检索说「退货 15 天」，两条都还在历史里；或者压缩摘要保留了旧结论，新检索给了新结论。

**自检**：在 dump 里搜同一主题，看是不是有两份打架的事实同时在场。

### 2.3 忽略指令 / 不守约束 → 指令被埋中间或被长历史稀释

你明明在 system 里写了「绝不能泄露内部价格」，模型还是泄露了。常见两因：(a) 这条指令被埋在一大段 system 中间（lost in the middle）；(b) 长对话历史把它「稀释」了——50 轮历史后，开头那条 system 约束的相对权重被冲淡。

**自检**：约束在 dump 里的位置靠不靠头尾？历史是不是长到把它淹了？

### 2.4 幻觉 / 编事实 → 检索没命中

模型一本正经胡说，往往是该提供的事实**根本没进上下文**——检索 query 改写偏了、向量召回 miss、或者召回了但被 token 预算截断。这不是模型「爱编」，是它手上没料。

**自检**：dump 里到底有没有那条该被引用的事实？没有 → 是检索问题，不是模型问题。

## 3. 症状 → 可能原因 → 对策速查表

| 症状 | 可能根因 | 排查动作 | 对策 |
|------|----------|----------|------|
| 答非所问 / 跑题 | 无关上下文稀释、context rot | dump 看有效信息占比 | 降检索 k、重排序、收紧召回阈值 |
| 前后矛盾 | context clash（冲突信息共存） | dump 搜同主题的多份事实 | 去重、保留最新、冲突时显式标注时效 |
| 忽略关键指令 | 指令埋中间 / 被长历史稀释 | 看约束位置 + 历史长度 | 约束放头尾、压缩历史、关键约束每轮重申 |
| 幻觉 / 编事实 | 检索 miss、被截断、query 改写偏 | dump 查事实在不在 | 修 query 改写、查召回率、放宽 token 预算 |
| 工具调不对 / 不调 | 工具定义被截断或描述不清 | dump 看 tools schema 全不全 | 裁冗余工具、改清描述、检查截断 |
| 输出格式时好时坏 | few-shot 被压缩挤掉、格式指令靠中间 | 看格式示例在不在 | 固定保留格式示例、指令放尾部 |
| 多轮越聊越离谱 | 历史污染累积、错误结论被反复引用 | dump 看历史里有没有错误结论沉淀 | 定期重置 / 摘要时剔除错误结论 |
| 偶发性翻车（同输入有时对有时错） | 缓存前缀变动、检索结果不稳定 | 对比两次 dump 的差异 | 固定检索、稳住缓存前缀 |

---

## 4. 标准排查流程

```
故障上报
  │
  ├─ 1. dump 模型实际收到的完整 messages + tools
  │
  ├─ 2. 关键信息在不在 dump 里？
  │       ├─ 不在 → 检索/拼装问题（query 改写、召回、截断）→ 修上游
  │       └─ 在 ↓
  │
  ├─ 3. 在哪个位置？是不是埋在中间？
  │       └─ 埋中间 → 重排，关键信息挪头尾
  │
  ├─ 4. 有没有矛盾信息共存？
  │       └─ 有 → 去重 / 保留最新 / 标时效（context clash）
  │
  ├─ 5. 无关噪声占比高不高？
  │       └─ 高 → 降 k / 重排序 / 收紧阈值（context rot）
  │
  └─ 6. 以上都正常 → 才轮到改 prompt 措辞 / 换模型
```

**注意流程顺序**：改措辞和换模型在**最后**。它们成本高、见效慢，而且大多数情况下治标不治本——上下文没拼对，换 Opus 也照样翻车。

## 5. 一个真实排查片段

症状：客服 Agent 偶尔把已经下线的旧政策当成现行政策回答。

```python
# 第一步：dump，在历史里搜「退货政策」
hits = [m for m in messages if "退货" in str(m.get("content", ""))]
for h in hits:
    print(h["role"], str(h["content"])[:80])
```

输出：

```
assistant 根据政策，退货期为 7 天 ...      ← 第 2 轮的旧检索结果，沉在历史里
user 现在还是 7 天吗？
assistant 退货期为 15 天（已更新）...      ← 第 6 轮的新检索结果
```

根因清楚了：**旧的 7 天结论留在历史里没清，和新的 15 天打架**（context clash + 历史污染）。模型在长历史里随机抓了旧的那条。对策：摘要历史时剔除已被新事实推翻的旧结论，或对带时效的检索结果显式标注「截至 X 日」让模型有取舍依据。**根因在上下文装载，一个字 prompt 都不用改。**

---

## 6. 排查心法

- ✅ 先 dump，后假设。没看到模型实际输入之前，一切都是猜
- ✅ 把「答非所问 / 矛盾 / 忽略指令 / 幻觉」对应到 rot / clash / 稀释 / miss 四类根因
- ✅ 改 prompt 和换模型放最后试
- ❌ 别一看到错就改措辞——大概率根因在那段你没打印的上下文里
- ❌ 别信「换个更强模型就好了」——上下文病，换药不换汤没用
