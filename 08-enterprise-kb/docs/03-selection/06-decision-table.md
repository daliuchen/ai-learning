# EKB 16：选型决策表——一页纸总结

> **一句话**：把第 03 章所有选型决策汇成一张可复用的表——每个决策对应「我们选了什么、为什么、什么情况下该选别的」。这页纸既是本项目的选型快照，也是你做自己项目时的选型 checklist。

## 1. 完整决策表

| 决策点 | 我们的选择 | 关键理由 | 什么时候该换 |
|--------|-----------|----------|--------------|
| 向量库 | pgvector | 几千~几万 chunk，少组件，能 join ACL | 千万级向量 → Qdrant/Milvus |
| Embedding | 可切换（中文优先 BGE） | 中文场景、保密、成本 | 中英混杂/无算力 → 商用 API |
| 生成框架 | Pydantic AI | 结构化输出强制引用+兜底 | 需动态编排/多 Agent → LangGraph |
| 编排方式 | 自写薄层（直线 pipeline） | 透明、可控、可逐步评估 | 出现动态决策信号 → 重框架 |
| 检索策略 | 混合 + rerank + 改写 | 解决口语 vs 术语错配 | 召回已够好可简化（先测） |
| 前端通信 | SSE | 服务器单向流式推送 | 需双向实时 → WebSocket |
| 后端语言 | Python (FastAPI) | 复用 AI 生态 | — |
| 前端语言 | TypeScript (Next.js) | UI 体验、组件生态 | — |
| 生成模型 | 强模型（Claude/GPT-4o 级） | 「找不到就承认」需指令遵循好 | 评估通过的更便宜模型 |
| 缓存 | prompt caching（模型层） | 降本，无需额外组件 | 出现 DB 瓶颈再加 Redis |

## 2. 选型背后的三条主线原则

这张表不是孤立决策的堆砌，背后有三条贯穿的原则：

### 原则一：匹配规模，不追最强

pgvector 而非 Milvus，自写薄层而非 LangGraph——都是「这个规模/复杂度用不到那么重」。**选型是匹配，不是攀比。**

### 原则二：少组件，少运维

能用 Postgres 解决就不引向量库，能用模型层缓存就不上 Redis。**每个组件都是长期负担。**

### 原则三：不确定就留可替换口子

embedding 做成接口、模型用 Pydantic AI 一行可换。**赌不准的地方留弹性，别焊死。**

## 3. 这套选型的「反面教材」长什么样

同一个需求，一个常见的过度工程版本：

```
❌ 过度工程版：
Milvus 集群（8000 向量）+ LangGraph 编排（直线 pipeline）
+ 独立 rerank 服务 + Redis + Kafka 文档变更队列
+ 微服务拆 5 个 + k8s 部署

结果：组件 8 个，运维爆炸，团队 3 人，光部署就两周，
      而真实负载一台机器一个 Postgres 就能扛。
```

对比我们的版本：**1 个 Postgres + 1 个 Python 服务 + 1 个前端**。功能一样，复杂度差一个数量级。这就是「先定边界再选型」省下来的东西。

## 4. 把这张表用到你自己的项目

做自己的 RAG 项目时，照着这个流程走一遍：

```
1. 先答 5 个边界问题（规模/形态/用户/更新/是否agentic）—— EKB 11
2. 规模 < 百万向量？ → pgvector，否则评估专用库
3. 中文为主？ → 优先评估 BGE，用自己的评估集实测
4. pipeline 是直线？ → 自写薄层 + Pydantic AI；是动态图 → 重框架
5. 每个「要不要上 X」都问「现在有这个瓶颈吗」 —— 没有就不上
6. 不确定的层 → 做成可替换接口
```

照这个走，大概率不会过度工程，也不会选漏关键能力。

## 5. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 抄别人的「最佳架构」 | 规模/场景不匹配 | 按自己边界推导 |
| 决策孤立做 | 整体过度/失衡 | 回到三条主线原则 |
| 一次定死所有选型 | 实测发现错了难改 | 不确定的留可替换 |
| 忽略运维成本 | 上线后维护痛苦 | 把「少组件」当硬指标 |
