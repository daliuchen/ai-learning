# EKB 15：什么时候才需要重框架——自写薄层的边界

> **一句话**：本项目大量「自己写薄薄一层」（检索、ingest），只在生成层用 Pydantic AI。这不是反框架，而是**按需用框架**——当逻辑是固定 pipeline 时，自写更透明可控；当逻辑需要动态编排、多 Agent、复杂状态时，重框架才回本。本篇给你判断的标准。

---

## 1. 自写 vs 框架，到底在权衡什么

```
自写薄层：              重框架（LangChain/LangGraph）：
+ 每行可读、可调试        + 现成的组件、集成、最佳实践
+ 没有黑盒               + 复杂编排开箱即用
+ 依赖少                 + 社区生态
- 要自己写胶水           - 抽象层多，出问题难定位
- 没有现成集成           - 版本升级可能破坏行为
                        - 简单场景上它是负担
```

核心权衡：**透明可控 vs 开箱即用**。场景越简单、越固定，自写越划算；场景越复杂、越动态，框架越回本。

---

## 2. 一条判断线：你的 pipeline 是「直线」还是「图」

| 你的逻辑长这样 | 选择 |
|----------------|------|
| 检索 → 生成（固定直线，无分支） | **自写薄层** |
| 检索 → 判断 → 可能再检索 → 调工具 → 反思 → 再生成（动态图） | 重框架（LangGraph 等） |

本项目的 Query 流是**一条直线**：`rewrite → 过滤 → 检索 → rerank → 生成`。没有「模型决定下一步做什么」的动态分支。直线 pipeline 自写最清楚——每一步就是一个函数，串起来就是流程。

---

## 3. 本项目「自写 vs 框架」的分工

我们不是全自写，也不是全框架，而是**按层分工**：

| 层 | 选择 | 为什么 |
|----|------|--------|
| Ingest（解析/分块/入库） | 自写 | 逻辑简单固定，自写最透明 |
| 检索（向量/BM25/混合/rerank） | 自写薄层 | 要精细控制和评估每一步，黑盒会挡路 |
| 生成（结构化输出） | Pydantic AI | 结构化输出是它的强项，自写校验/重试不值得 |
| 前端 | Next.js + 组件库 | UI plumbing 借现成的 |

原则：**无聊且通用的借脚手架，核心且要可控的自写，有专门强项的库用库。**

---

## 4. 什么信号说明「该上重框架了」

如果将来需求变了，出现下面这些信号，就该考虑 LangGraph 这类编排框架：

- **动态决策**：模型要自己决定「先检索还是先问澄清问题」
- **多 Agent 协作**：检索 Agent + 写作 Agent + 审核 Agent 互相传递
- **复杂状态/循环**：「检索不满意就改写 query 重检索」的反思循环
- **需要中断/恢复**：长任务要能暂停、人工介入、再继续

这些是 [01 手册 LangGraph](/docs/01-langchain/03-langgraph/01-introduction) 的主场。**本项目刻意不触发这些信号**（见非目标），所以不需要。

---

## 5. 自写不等于从零造轮子

「自写薄层」常被误解为「什么都自己实现」。不是。我们仍然用：

- `psycopg` / `pgvector` 连数据库（不自己写 SQL 协议）
- `rank-bm25` 算 BM25（不自己实现算法）
- `sentence-transformers` 做 embedding/rerank（不自己训模型）
- Pydantic AI 做结构化生成

「自写薄层」指的是**把这些库串成业务流程的那层胶水自己写**，而不是引一个大框架替你串、然后你看不清它怎么串的。**用库，但不被框架接管控制流。**

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 简单直线 pipeline 上 LangGraph | 复杂度白增，调试难 | 自写函数串起来 |
| 「自写」理解成全造轮子 | 重复实现 BM25/embedding | 用库做底层，自写胶水 |
| 怕麻烦全交给大框架 | 出问题定位困难 | 核心可控部分自写 |
| 该上框架时还硬撑自写 | 动态编排代码失控 | 出现动态/多 Agent 信号就上框架 |
