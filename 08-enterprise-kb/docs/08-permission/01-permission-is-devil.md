# EKB 38：权限是魔鬼——企业知识库区别于玩具的核心

> **一句话**：一个不管权限的 RAG demo，谁都能做；一个能保证「HR 薪资文档绝不被普通员工检索到」的系统，才是企业知识库。权限不是事后加的功能，而是从数据模型到检索流程都要贯穿的设计。本篇讲清为什么权限这么难、错在哪会泄漏。

---

## 1. 为什么权限是企业知识库的分水岭

```
玩具 RAG demo：所有文档对所有人可见 → 检索全库 → 答
企业知识库：    HR 文档只 HR 看、法务文档只法务看、未发布产品只产品组看
              → 必须按提问者身份限定可检索范围
```

没有权限，知识库就**不能上线**——把薪资细则、绩效标准、未发布规划答给所有人，是数据事故。所以权限是企业知识库的**准入门槛**，不是加分项。

---

## 2. 泄漏会从哪发生：四个隐蔽入口

权限难，是因为泄漏点很隐蔽，不只一处：

| 泄漏入口 | 怎么漏 |
|----------|--------|
| 检索召回 | 无权文档进了候选集 → 即使没展示，也进了 prompt |
| Prompt 拼接 | 召回时过滤了，但日志/缓存里留了全量 |
| 引用展示 | 答案过滤了，但引用列表泄漏了无权文档标题 |
| 模型记忆/串话 | 多租户共享 context，A 的内容串到 B |

最隐蔽的是第一个：**很多人以为「答案里没提到就没泄漏」，但只要无权文档进了 prompt，模型就可能在答案里复述其内容**——哪怕你没在引用里列它。所以过滤必须在**召回阶段**，不是展示阶段。

---

## 3. 核心原则：在检索阶段过滤，不在生成/展示阶段

```
❌ 错误（晚过滤）：
检索全库 → 召回含无权文档 → 喂给模型 → 答案/引用里再过滤掉无权的
   问题：无权内容已进 prompt，可能被复述；日志也记了

✅ 正确（早过滤）：
按角色限定范围 → 只召回有权文档 → 无权文档从未进入候选
   保证：模型根本看不到无权内容
```

这是整章最重要的一句话：**无权文档应该从一开始就不进候选集，而不是进了再藏。** 详见 [03-filter-at-retrieval](./03-filter-at-retrieval.md)。

---

## 4. 权限模型要多复杂

按需求选最简够用的模型，别过度设计：

| 模型 | 表达力 | 适合 |
|------|--------|------|
| 公开/不公开（单 flag） | 最弱 | 对外客服 |
| **基于角色（RBAC）** | 中 | 大多数企业知识库 ✅ |
| 基于属性（ABAC） | 强 | 复杂合规场景 |
| 行级/字段级 | 最强 | 极敏感数据 |

本项目用 **RBAC**（角色）——员工有角色（engineer/hr/finance...），文档标可见角色，匹配即可见。够覆盖「HR 文档只 HR 看」这类需求，又不至于像 ABAC 那样复杂。这是第 02 章数据模型 `acl(doc_id, role)` 的依据。

---

## 5. 权限和检索质量的张力

加权限过滤会**缩小检索范围**，这带来一个副作用：

```
普通员工提问 → 范围缩小到「公开文档」→ 候选变少
→ 如果某问题的答案在受限文档里 → 该员工就是查不到（这是对的！）
→ 但要确保：是「无权」而非「检索 bug」导致的查不到
```

所以权限上线后，评估要**区分两种「查不到」**：无权（正确，应兜底）vs 检索失败（bug，要修）。这就是为什么评估用例要标 `asker_role`（[04-eval/02](../04-eval/02-build-testset.md)）——同一问题不同角色，期望结果不同。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 权限当事后功能 | 数据模型推翻重做 | 设计期就贯穿 |
| 检索后/展示时才过滤 | 无权内容进了 prompt，可能被复述 | 召回阶段过滤 |
| 只过滤答案，不过滤引用 | 引用泄漏无权文档标题 | 引用也按权限过滤 |
| 权限模型过度设计 | 复杂难维护 | RBAC 够用就别上 ABAC |
| 不区分「无权」和「检索失败」 | 误判 bug 或漏洞 | 评估标 asker_role 区分 |
