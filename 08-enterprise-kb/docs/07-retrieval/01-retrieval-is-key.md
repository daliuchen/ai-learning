# EKB 32：检索是命门——为什么 RAG 的上限由检索决定

> **一句话**：RAG = 检索 + 生成，但**检索决定上限**。模型再强，如果该用的片段没被检索到，它要么答不出、要么编。企业知识库最大的检索痛点是「员工口语 vs 文档术语」的错配。本章一项项加增强（BM25、混合、rerank、改写）来打通这个错配，每项都用评估量化。

---

## 1. 一个残酷的事实：检索漏了，生成无力回天

```
用户问「钱能不能要回来」
文档里写的是「退款流程」
向量检索：「钱」和「退款」语义相关，可能召回；但「要回来」是口语，召回排名靠后
→ 退款文档没进 top-5
→ 模型拿到的全是无关片段
→ 要么答不出，要么用其它片段瞎拼
```

注意：**这不是模型的错**。GPT-4o、Claude 再聪明，也只能基于「你给它的片段」回答。片段错了，神仙难救。所以优化 RAG，**先优化检索，再优化生成**。

---

## 2. 企业知识库的三类检索错配

员工的问法和文档的写法，错配主要有三种：

| 错配类型 | 例子 | 哪种检索能救 |
|----------|------|--------------|
| 同义/口语 | 「钱要回来」vs「退款」 | query 改写 + 向量 |
| 专有名词/缩写 | 「OA 怎么登」——OA 是专名 | BM25 关键词 |
| 表述距离远 | 「新人第一天干啥」vs「入职流程」 | query 改写 |

单一的向量检索对**同义**还行，但对**专有名词/缩写**经常失手（embedding 对罕见专名不敏感），对**口语化**也容易漏。这就是为什么要混合检索 + 改写。

---

## 3. 本章的增强路线图

我们在 MVP 单路向量检索的基础上，按顺序加四项，每项独立评估：

```
MVP: 单路向量          recall@5 = 0.66 (baseline)
 │
 ├─ + BM25 关键词       → 专名/缩写召回改善
 ├─ + 混合融合（RRF）    → 两路优势结合
 ├─ + Rerank 精排       → 头部排序质量
 └─ + Query 改写        → 口语/远表述召回
                        recall@5 = ? （逐步逼近 0.85+）
```

每加一项都跑 `run_eval`，记录 delta。**一次只加一项**——这样才知道每项值多少（[01-intro/04](../01-intro/04-mvp-first.md) 的纪律）。

---

## 4. 召回与精排：两个不同的目标

理解检索要分两个阶段，目标相反：

```
召回阶段（recall 优先）：宁可多召回，别漏。k 设大（如 20）
   → BM25 + 向量两路，各召回 20 条，合并
精排阶段（precision 优先）：从召回里挑最相关的少数。
   → rerank 排序，取 top 3-5 进 prompt
```

新手常犯的错：检索阶段就只取 top-5。这样召回窗口太窄，相关片段排第 7 就被丢了。正确做法：**召回宽（20）→ 精排窄（5）**。详见 [04-rerank](./04-rerank.md)。

---

## 5. 检索增强的「天花板」与「地板」

加增强前，先认识到检索质量有边界：

- **地板**：再差的检索，至少有距离阈值兜底（不会拿一堆垃圾硬答）
- **天花板**：受限于**分块质量**和**数据本身**——如果文档里压根没写这个知识，检索再强也召回不到（这时该走兜底，或提示补文档）

所以检索增强不是万能药。如果评估发现某些问题 recall 始终是 0，先查：是检索不行，还是**知识库里根本没有**？后者要回到 ingest/补文档，不是调检索。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 只优化生成（换大模型）不优化检索 | 检索漏了白搭 | 先优化检索 |
| 召回阶段就取 top-5 | 相关片段被早早丢弃 | 召回宽、精排窄 |
| 一次加多项增强 | 归因不了 | 一次一项，跑评估 |
| 把所有问题都归咎于检索 | 其实是缺文档 | 区分「检索差」vs「没数据」 |
| 不设地板（阈值） | 垃圾片段进生成 | 距离阈值兜底 |
