# EKB 47：Prompt Caching——降本的第一招

> **一句话**：知识库问答的 prompt 里，系统提示和高频文档片段是**反复出现**的。用 prompt caching 把这些固定前缀缓存住，重复请求只为「变化的部分」付费，能显著降本降延迟。本篇讲怎么组织 prompt 才能吃满缓存。

---

## 1. 哪些部分是可缓存的

一次问答的 prompt 大致是：

```
[系统提示]        ← 每次都一样，可缓存
[（可选）高频文档]  ← 热门问题命中相同片段，部分可缓存
[检索到的片段]     ← 随问题变，不可缓存
[用户问题]         ← 每次都变，不可缓存
```

prompt caching 的原理：**相同的前缀**只需计算一次，后续命中缓存的请求跳过这部分的计算/计费。所以关键是**把固定不变的内容放在 prompt 最前面**（[07 手册 prompt caching](/docs/07-context-engineering/07-long-context/03-prompt-caching)）。

---

## 2. 组织 prompt 让前缀稳定

```python
# ❌ 固定内容散在中间，缓存命中率低
prompt = f"{retrieved_chunks}\n\n规则：只用文档回答...\n问题：{q}"

# ✅ 固定前缀在最前，变化的在后
SYSTEM = "你是企业知识库助手。只能基于文档片段回答..."   # 长且固定 → 缓存
prompt = f"{retrieved_chunks}\n\n问题：{q}"               # 变化部分在后
```

Anthropic 的 Claude 用 `cache_control` 标记缓存断点，OpenAI 自动缓存长前缀。要点一致：**让不变的内容成为稳定前缀**。系统提示越长越固定，缓存收益越大。

```python
# Claude 显式标记缓存（示意）
messages=[{
    "role": "system",
    "content": [{"type": "text", "text": SYSTEM,
                 "cache_control": {"type": "ephemeral"}}],   # 缓存系统提示
}]
```

---

## 3. 知识库场景的缓存机会

| 可缓存对象 | 怎么吃缓存 | 收益 |
|-----------|-----------|------|
| 系统提示 | 始终放最前、内容固定 | 每次请求省系统提示的 token |
| 高频文档全文 | 极小知识库可全量缓存（见下） | 跳过 RAG |
| 改写器的系统提示 | query 改写器的指令固定 | 改写调用降本 |

**注意**：检索到的片段**每次不同**，通常不可缓存——这是 RAG 和「全量长上下文」的区别。RAG 缓存的主要是系统提示和改写器指令，单次省的不算巨大，但高 QPS 下累积可观。

---

## 4. 小知识库的特例：全量缓存替代检索

如果知识库特别小（[12 篇](../03-selection/02-vector-db-pgvector.md) 提过），可以反过来——**不检索，把全部文档塞进可缓存前缀**：

```
[系统提示 + 全部文档]   ← 一次性缓存住（几万 token）
[用户问题]              ← 每次只付问题的 token
```

适用条件：文档总量 < 窗口一半、更新不频繁（更新会让缓存失效）。这时候连向量库都省了。但企业知识库通常超出这个量级，所以本项目主线还是 RAG + 缓存系统提示。**这是个要按规模判断的取舍，不是非此即彼。**

---

## 5. 缓存失效要心里有数

缓存不是永久的，几个失效点要知道：

- **TTL 过期**：缓存有生存时间（如 5 分钟），低频请求可能每次都 miss
- **前缀变了**：系统提示改一个字，整个缓存失效——所以别频繁改 prompt
- **文档更新**：如果缓存了文档全文，文档一改缓存就失效

实践建议：**把系统提示稳定下来**（别动不动改措辞），让缓存命中率高。频繁 A/B 测 prompt 的阶段，缓存收益会打折——那是评估迭代期的正常代价。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 固定内容放 prompt 中间 | 缓存命中率低 | 不变的放最前 |
| 频繁改系统提示 | 缓存反复失效 | 稳定后再上缓存 |
| 指望检索片段也被缓存 | 片段每次变，缓存不了 | 缓存系统提示为主 |
| 小知识库还硬走 RAG | 错过全量缓存的简化 | 按规模判断 |
| 不监控缓存命中率 | 不知道有没有省到 | 观测缓存命中（下章） |
