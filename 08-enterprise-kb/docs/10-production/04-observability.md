# EKB 50：可观测性——看得见每一次问答里发生了什么

> **一句话**：RAG 系统出问题时，「答得不好」只是表象，你需要看到链路里每一步：检索到了什么、rerank 怎么排的、prompt 长什么样、模型怎么答的。本篇讲该记录哪些信号、怎么做链路追踪，让系统从黑盒变成可观测。

---

## 1. 为什么 RAG 特别需要可观测

```
用户报告：「问 X 答得不对」
没有可观测：你只能干瞪眼，复现都难
有可观测：调出这次问答的完整链路——
  → 哦，检索根本没召回对的文档（检索问题）
  → 或者召回对了但 rerank 排到第 8（排序问题）
  → 或者片段对了但模型没用上（生成问题）
```

RAG 链路长、每步都可能出错，**没有可观测就没法定位问题**。这是从 demo 走向生产最关键的基建之一。

---

## 2. 每次问答要记录什么

按链路逐步记录，存进 `query_logs` 或专门的 trace：

```python
trace = {
    "query_id": ...,
    "question": q,
    "user_role": role,
    # 检索层
    "rewritten_queries": [...],          # 改写成了啥
    "retrieved": [                        # 召回了哪些，带分数
        {"chunk_id": .., "doc_id": .., "vector_dist": .., "bm25": .., "rrf": ..},
    ],
    "reranked": [{"chunk_id": .., "rerank_score": ..}],   # rerank 后
    # 生成层
    "prompt_tokens": .., "completion_tokens": ..,
    "found": True, "cited_doc_ids": [..],
    # 性能
    "latency": {"rewrite": .., "retrieve": .., "rerank": .., "generate": ..},
    # 反馈（事后回填）
    "feedback": None,
}
```

有了这份 trace，任何一次问答都能完整回放——这是分诊（[49 篇](./03-feedback-loop.md)）和排障（下篇）的基础。

---

## 3. 该看的几个核心指标

把单次 trace 聚合成监控面板，盯这几个：

| 指标 | 看什么 | 异常信号 |
|------|--------|----------|
| found=false 率 | 多少问题答不出 | 突然升高 = 检索退化或知识盲区 |
| 平均 rerank top1 分 | 检索质量趋势 | 下降 = 检索变差 |
| TTFT / 总延迟 P95 | 性能 | 超预算（[48篇](./02-cost-latency.md)） |
| token 用量 | 成本 | 突增 = 片段塞太多 |
| 👎 率 | 用户满意度 | 上升要排查 |
| 越权召回数 | 安全红线 | **任何 >0 立即告警** |

越权召回数是**红线监控**——它在生产里也要持续盯，不只是上线前测一次。

---

## 4. 用现成的 LLM 可观测工具

不用全自己造。LLM 应用可观测已有成熟工具，能自动记录链路、token、延迟：

- **LangSmith**（[01 手册](/docs/01-langchain/02-langsmith/01-introduction)）：trace + 评估一体
- **Langfuse**：开源，自托管友好
- **Pydantic Logfire**：和 Pydantic AI 同源，集成顺

它们能把上面的 trace 自动可视化，省去自己写日志面板。Pydantic AI 项目用 Logfire 几乎零成本接入：

```python
import logfire
logfire.configure()
logfire.instrument_pydantic_ai()    # 自动 trace 每次 agent 调用
```

自己造还是用工具，看团队——但**「能回放每次问答」这个能力必须有**，工具只是手段。

---

## 5. 日志里的隐私与权限

可观测会记录问题和文档内容，这本身有合规考虑：

- **敏感内容**：trace 里会有受限文档片段，trace 存储也要控制访问权限
- **PII**：用户问题可能含个人信息，按合规要求脱敏/保留期管理
- **不要把受限内容泄漏给看日志的人**：运维看 trace 时，受限文档内容是否该脱敏？

**可观测不能成为权限的后门**——你费力做的检索层权限过滤，别在日志这里漏出去。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 只记问题和答案 | 中间环节是黑盒，没法定位 | 记录全链路 trace |
| 出问题无法复现 | 干瞪眼 | trace 支持回放 |
| 越权召回不在线监控 | 漏洞上线后才发现 | 红线持续告警 |
| 全自己造日志 | 重复造轮子 | 用 LangSmith/Logfire |
| 日志泄漏受限内容 | 权限后门 | trace 也要控权限/脱敏 |
