# EKB 49：反馈闭环——让知识库持续变好

> **一句话**：知识库上线不是终点，是起点。用户的「有用/没用」「引用不准」反馈，加上 `query_logs` 里答不好的真实问题，是优化的金矿。本篇讲怎么收集反馈、把它变成评估用例和优化动作，形成「上线 → 反馈 → 改进 → 再上线」的循环。

---

## 1. 闭环长什么样

```
用户提问 → 系统回答 → 用户反馈（👍/👎/引用不准）
   │                          │
   │                          ▼
   │                    写入 query_logs
   │                          │
   │         ┌────────────────┴────────────────┐
   │         ▼                                  ▼
   │   答不好的问题                        引用被标错的
   │         │                                  │
   │         ▼                                  ▼
   │   补进评估集 / 补文档                调检索/分块
   │         │                                  │
   └─────────┴──────── 改进后重新评估 ──────────┘
```

核心：**真实使用产生的数据，反哺系统改进**。没有这个闭环，系统就停在上线那天的水平。

---

## 2. 收集哪些反馈信号

| 信号 | 来源 | 说明什么 |
|------|------|----------|
| 👍/👎 | 用户点 | 答案整体满不满意 |
| 引用不准 | 引用卡片按钮（[44篇](../09-frontend/02-citation-clickback.md)） | 检索/排序问题 |
| 追问/重问 | query_logs 里同一人连续问相似问题 | 上一次没答好 |
| found=false 率 | 系统记录 | 哪些问题知识库没覆盖 |
| 零反馈但低相关度 | 检索分数低 | 潜在答不好 |

`query_logs` 表（[09 篇](../02-design/04-data-model.md)）已经存了问题、检索结果、答案、found、feedback——这些字段就是闭环的原料。

---

## 3. 从反馈到行动：分诊

不同信号对应不同的修法，别一锅炖：

```python
def triage_feedback(log_row) -> str:
    if not log_row["found"]:
        return "缺文档"          # 知识库没这内容 → 补文档
    if log_row["feedback"] == -1 and log_row["retrieved_ids"]:
        # 召回了但答不好 → 看是检索排序问题还是生成问题
        return "检索或生成"
    if log_row["bad_citation"]:
        return "检索排序"        # 引用错 → rerank/分块
    return "ok"
```

| 分诊结果 | 行动 |
|----------|------|
| 缺文档 | 通知文档负责人补充，加 ingest |
| 检索排序 | 看 rerank/分块，补进评估集调优 |
| 生成问题 | 看 prompt，片段对但答歪了 |

**关键**：先判断是哪一层的问题（检索 vs 生成 vs 缺数据），再动手——和评估的分层思路一致（[04-eval/01](../04-eval/01-why-eval-first.md)）。

---

## 4. 把真实问题补进评估集

闭环最有价值的产出：**用真实失败案例扩充评估集**。

```python
# 从日志里捞答不好的，人工核对后加进 testset
def harvest_eval_cases(conn):
    bad = conn.execute(
        "SELECT question, user_role, retrieved_ids, found "
        "FROM query_logs WHERE feedback = -1 "
        "ORDER BY created_at DESC LIMIT 50").fetchall()
    # 人工核对：这个问题正确答案在哪篇文档？标成新评估用例
    return bad
```

这样评估集从「上线前人工想的 20 条」成长为「反映真实使用的几百条」，越来越能代表真实分布。**评估集是活的**（[18 篇](../04-eval/02-build-testset.md)），闭环就是它成长的养分。

---

## 5. 闭环的节奏

不用实时，定期跑就好：

```
每周：看 👎 率、found=false 率、bad_citation 率的趋势
每周：捞 top 失败问题，分诊 → 补文档 / 补评估用例
每月：用扩充后的评估集重新评估，看整体指标趋势
按需：指标下滑时，回到对应环节优化
```

形成节奏后，知识库会**越用越准**——这是它和「上线即巅峰」的静态系统的本质区别。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 没有反馈入口 | 不知道哪答不好 | 👍/👎 + 引用反馈 |
| 收集了不分诊 | 不知道修哪层 | 先分诊再行动 |
| 不把真实问题补进评估集 | 评估集脱离实际 | 持续 harvest |
| found=false 不跟进 | 知识盲区一直在 | 通知补文档 |
| 闭环没节奏 | 反馈堆着不处理 | 周/月固定复盘 |
