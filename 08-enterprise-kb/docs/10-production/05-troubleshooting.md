# EKB 51：常见故障排查——RAG 出问题怎么定位

> **一句话**：「答得不对」是症状，不是病因。本篇给一套排查决策树——从症状出发，借助可观测的 trace，逐步定位到底是检索、生成、数据还是权限的问题，并给出对应修法。这是把前面所有章节串起来用的「急诊手册」。

---

## 1. 排查的第一步：分层定位

RAG 出错，先用 trace 把问题归到某一层（呼应评估的分层思路）：

```
症状：答得不对
  │
  ├─ 看 trace：相关文档在召回结果里吗？
  │   ├─ 不在 → 【检索层问题】
  │   └─ 在 → 看 rerank 排序
  │        ├─ 排得很后/被裁掉 → 【排序问题】
  │        └─ 进了 prompt → 【生成层问题】
  │
  └─ trace 显示 found=false 但其实库里有 → 【阈值/检索问题】
```

**先定位层，再修**——不要一上来就调 prompt，可能根本是检索没召回。

---

## 2. 症状 → 病因 → 修法对照表

| 症状 | 可能病因 | 怎么确认 | 修法 |
|------|----------|----------|------|
| 该有的答案答不出 | 检索没召回 | trace 看召回里有没有目标 doc | 加改写/混合/调 chunk |
| 答案张冠李戴 | 召回了无关片段且排前 | 看 rerank 分数 | 调 rerank / 收紧阈值 |
| 模型编了个答案 | 兜底失守 | 看 found 和片段相关度 | 强化三道防线（[30篇](../06-basic-rag/04-say-i-dont-know.md)） |
| 引用指向错文档 | rerank 排序/分块差 | 看 cited vs 实际相关 | 调 rerank / 重切分块 |
| 答案不全 | 片段被切断/召回不全 | 看 chunk 边界 | 调分块大小/重叠 |
| 专名查不到 | 向量对专名弱 | 看 BM25 路有没有命中 | 确认 BM25 + 专名词典 |
| 口语问题查不到 | 表述距离远 | 看改写有没有生效 | 检查 query 改写 |
| 越权看到文档 | 权限过滤漏 | leak_scan | 修过滤（[40篇](../08-permission/03-filter-at-retrieval.md)） |
| 延迟突然变高 | 某环节退化 | 看各步 latency | 定位慢的环节 |

## 3. 检索层问题的细分排查

如果定位到「检索没召回」，再往下分：

```
召回不到目标文档？
  ├─ 文档真的 ingest 了吗？      → 查 chunks 表有没有这文档的 chunk
  ├─ embedding 是不是坏了？      → 查向量是不是全 0 / 维度对不对
  ├─ 是不是被权限过滤掉了？      → 用 all 角色重试，看是不是权限问题
  ├─ 是专名/口语错配吗？         → 看 BM25 和改写有没有兜住
  └─ 知识库里压根没有？          → 这不是 bug，该走兜底 / 补文档
```

最后一条最容易被误判成 bug——**库里没有的内容，检索再强也召回不到**。先确认数据存在，再怀疑检索。

---

## 4. 生成层问题的细分排查

如果「片段对了但答歪了」：

```
片段进了 prompt 但答案不对？
  ├─ 片段里真的有答案吗？        → 人读一遍片段，确认信息在
  ├─ 模型忽略了片段用了常识？     → 强化系统提示「只用片段」
  ├─ 片段太多噪声淹没了？        → 减少 top-k，rerank 更狠
  ├─ prompt 拼接格式乱？         → 检查 format_chunks
  └─ 模型能力不够？             → 换更强模型试
```

## 5. 一个真实排查示例

```
报告：「问『OA 怎么改密码』答不出，但确实有这文档」

1. 看 trace → 召回结果里没有 OA 文档 → 检索层问题
2. 查 chunks → OA 文档的 chunk 存在 ✅，排除「没 ingest」
3. 用 all 角色重试 → 还是召回不到 → 排除权限
4. 看两路：向量召回排第 30，BM25 也很后
5. 发现：「OA」被 jieba 切成了「O」「A」→ BM25 匹不中
6. 修：jieba.add_word("OA")，重建 BM25 索引
7. 重试 → OA 文档召回到 top-3 ✅
8. 补一条评估用例防回归
```

这个流程展示了「症状 → trace → 逐层排除 → 定位根因 → 修 → 补回归用例」的完整闭环。**每次排障都该以「补一条评估用例」收尾**，防止同样的问题再来。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 一出错就调 prompt | 可能根本是检索问题 | 先分层定位 |
| 把「缺数据」当 bug 修 | 白费功夫 | 先确认数据存在 |
| 修完不补回归用例 | 同问题反复出现 | 每次以补用例收尾 |
| 没 trace 靠猜 | 定位慢、修错 | 依赖可观测 |
| 只修单例不看面 | 同类问题还在 | 看是否一类问题 |
