# EKB 52：上线 Checklist——从能跑到敢上线

> **一句话**：系统功能完整、能跑通，不等于敢上线。本篇给一份覆盖「质量、安全、性能、运维、闭环」五个维度的上线检查清单，逐条过关才发布。把它当门禁——尤其安全红线，一条不过就不上线。这也是整本手册的收口。

---

## 1. 质量门禁

指标达标才发布，标准来自评估集（[04 章](../04-eval/01-why-eval-first.md)）：

- [ ] recall@5 ≥ 目标值（如 0.80）
- [ ] 引用准确率 ≥ 目标值（如 0.85）
- [ ] **兜底正确率 ≥ 0.95**（答不出时不编，权重最高）
- [ ] 各类型问题（直球/口语/专名/跨文档）都达标，不被平均分掩盖
- [ ] 评估集覆盖了主要文档空间和角色

```python
def quality_gate(rows):
    assert recall_at_5(rows) >= 0.80
    assert citation_accuracy(rows) >= 0.85
    assert fallback_accuracy(rows) >= 0.95     # 不编答案
    print("✅ 质量门禁通过")
```

---

## 2. 安全红线（一条不过就不上线）

这些是不可妥协的：

- [ ] **越权召回 = 0**：`leak_scan` 穷举扫描全绿（[42篇](../08-permission/05-no-leak-verify.md)）
- [ ] 引用层不泄漏受限文档
- [ ] 角色来自服务端可信来源，前端无法伪造
- [ ] BM25 和向量两路都验证过权限
- [ ] 多角色用户、改可见性场景都测过
- [ ] 日志/trace 不泄漏受限内容

```python
def security_gate(conn):
    assert leak_scan(conn) == [], "越权泄漏，禁止上线！"
    # 红线：不通过直接 fail，不存在「先上线再修」
```

**质量可以渐进改善，安全红线不行**——越权泄漏一次就是事故。

---

## 3. 性能门禁

- [ ] TTFT P95 < 目标（如 1.5s）
- [ ] 总延迟 P95 < 目标（如 4s）
- [ ] 单次问答成本 < 预算（如 $0.01）
- [ ] prompt caching 命中率达标
- [ ] 压测过预期并发

---

## 4. 运维就绪

- [ ] 可观测接好：能回放任意一次问答的完整链路（[50篇](./04-observability.md)）
- [ ] 核心指标有监控面板（found率/延迟/成本/👎率/越权数）
- [ ] 越权召回数有实时告警
- [ ] 文档增量更新流程跑通（[26篇](../05-ingest/06-incremental.md)）
- [ ] 数据库备份、向量索引可重建
- [ ] 模型 API 失败有降级（超时/限流的 fallback）

---

## 5. 闭环就绪

- [ ] 反馈入口上线（👍/👎/引用不准）
- [ ] `query_logs` 记录完整
- [ ] 定好复盘节奏（周看趋势、月重评估）
- [ ] 从日志 harvest 评估用例的流程跑通（[49篇](./03-feedback-loop.md)）

---

## 6. 把门禁接进 CI

上面的门禁不是上线前手动跑一次，而是**每次改动自动跑**，防回归：

```python
# CI: 任何 PR 合并前必跑
def launch_gate(conn):
    rows = run_eval()
    quality_gate(rows)        # 质量达标
    security_gate(conn)       # 安全红线（最重要）
    perf_gate()               # 性能达标
    print("✅ 全部门禁通过，可发布")
```

安全门禁尤其要常驻 CI——重构检索、改缓存、调融合，都可能无意中破坏权限。**门禁红就不准合并。**

## 7. 全书回顾：我们做了什么

从「读完手册≠会落地」出发，这本手册带你把前 7 本的零件，拼成了一个真实系统：

```
选型方法论（不追最强，匹配规模）
→ 评估先行（立标尺，不闭眼改）
→ Ingest（语义分块带元数据）
→ 基础 RAG（结构化输出 + 引用 + 兜底）
→ 检索增强（混合 + rerank + 改写，每步量化）
→ 权限（检索阶段过滤，越权零泄漏）
→ 前端（流式 + 引用回链 + TS/Python 拆分）
→ 生产化（降本、可观测、排障、门禁）
```

核心心法贯穿始终：**先定边界再选型、评估驱动一切、安全是红线、闭环让系统越用越好**。这些不只适用于知识库——是任何 LLM 应用从 demo 到生产都要走的路。

---

## 8. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 能跑就上线 | 质量/安全无保障 | 过完五维门禁 |
| 安全红线「先上后修」 | 泄漏即事故 | 红线不过不上线 |
| 门禁只上线前跑一次 | 后续改动破坏 | 接进 CI 常驻 |
| 没闭环就上线 | 上线即巅峰 | 反馈机制先就位 |
| 只看平均指标 | 薄弱环节漏网 | 分类型+红线都查 |
