# EKB 02：最终成品长什么样

> **一句话**：我们要做一个**对内**的企业知识库问答系统——员工带着角色提问，系统只检索他**有权限看**的文档，用混合检索 + rerank 找出最相关片段，让模型生成**带引用、答不出就承认**的回答，引用能点回原文。

---

## 1. 一次完整的问答，发生了什么

先看成品的一次交互，建立全局印象。员工小王（角色：`engineer`）在问答页输入：

> 「公司的报销额度是多少？」

系统内部走了这几步：

```
1. 收到问题 + 当前用户角色 = engineer
2. Query 改写：「报销额度」→ 扩展出「报销 限额 标准 费用」等
3. 权限过滤：只在 engineer 可见的文档里检索（HR 内部薪资文档被排除）
4. 混合检索：BM25 关键词 + 向量语义，各召回 20 条
5. 融合 + Rerank：精排出最相关的 4 条片段
6. 生成：把 4 条片段 + 问题喂给模型，强制输出「答案 + 引用了哪几篇」
7. 流式返回：答案逐字蹦出，下方列出引用的文档标题（可点回原文）
```

返回给小王的是：

> **答案**：差旅报销单次上限 2000 元，需在出差结束后 15 个工作日内提交。
> **引用**：📄《差旅与报销制度》第 3 节 · 📄《费用报销操作指南》

如果问的是「公司今年裁员计划」这种**没有对应文档**的问题，系统会回答：

> 抱歉，我在你有权限访问的知识库里没有找到相关信息。

——**不编**。这是企业知识库的底线。

## 3. 三个让它「不是玩具」的特性

普通 RAG demo 和这个项目的区别，就在这三点：

### 3.1 权限（最核心）

每篇文档标注「哪些角色可见」。检索时**先按角色过滤**，HR 的薪资文档对普通工程师**根本不会被召回**——不是召回了再藏起来，而是从一开始就不进候选集。详见第 08 章。

### 3.2 引用溯源

每个答案都附带「引用了哪几篇文档」，可点回原文。企业场景里，**没引用的答案没人敢信**。这个特性反推了数据模型：chunk 必须存「属于哪篇文档、文档的标题和链接」。详见 [06-basic-rag/03-citation](../06-basic-rag/03-citation.md)。

### 3.3 答不出就承认

检索不到相关内容时，明确说「没找到」，而不是用模型的参数化知识硬编一个。**宁可漏，不可错**。详见 [06-basic-rag/04-say-i-dont-know](../06-basic-rag/04-say-i-dont-know.md)。

## 4. 技术栈一览（细节见下一篇）

| 层 | 选择 | 为什么 |
|----|------|--------|
| 向量库 | pgvector | 几千~几万 chunk，不需要重型库 |
| 检索 | 混合（BM25 + 向量）+ rerank | 解决「员工口语 vs 文档术语」错配 |
| 生成框架 | Pydantic AI | 结构化输出强制「答案 + 引用」 |
| Embedding | 可切换（中文优先） | 当 06 手册的活教材 |
| 前端 | Next.js + SSE | 流式回答 + 引用卡片 |
| 后端 | FastAPI | 对齐 6 本 Python 手册 |

## 5. 我们会分几步搭出来

不是一次性写完，而是**每一步都能独立跑通**、且能量化「这步带来多少提升」：

| 阶段 | 产出 |
|------|------|
| 评估先行（04 章） | 一个能打分的测试集——后面每次改动都有标尺 |
| Ingest（05 章） | demo 文档 → 入库，可以查到 chunk |
| 基础 RAG（06 章） | 端到端跑通：能问能答带引用 |
| 检索增强（07 章） | 加 BM25/rerank/改写，准确率肉眼可见地涨 |
| 权限（08 章） | 越权文档不召回，多角色验证通过 |
| 前端（09 章） | 能在浏览器里用的问答页 |
| 生产化（10 章） | 降本、兜底、反馈闭环、上线 checklist |
