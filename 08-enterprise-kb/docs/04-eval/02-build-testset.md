# EKB 18：构建评估测试集

> **一句话**：评估集就是一堆「问题 → 期望命中的文档（+ 期望答案要点）」。本篇教你怎么从零造出第一版 20 条，覆盖哪些类型，以及为什么要标注「以什么角色提问」。这是后续所有迭代的标尺。

---

## 1. 一条评估用例长什么样

最小结构：每条记录一个问题、它**应该**命中哪些文档、以及（可选）答案要点：

```jsonl
{"id": 1, "question": "年假有几天怎么申请？", "expected_doc_ids": [12], "asker_role": "all", "answer_points": ["按工龄1-5年5天", "OA系统提交"]}
{"id": 2, "question": "差旅报销单次上限多少？", "expected_doc_ids": [7, 9], "asker_role": "all", "answer_points": ["2000元", "15个工作日内"]}
{"id": 3, "question": "VPN连不上怎么办？", "expected_doc_ids": [21], "asker_role": "engineer", "answer_points": ["重启客户端", "联系IT"]}
{"id": 4, "question": "公司明年的薪资普调计划？", "expected_doc_ids": [], "asker_role": "all", "answer_points": ["应答：未找到"]}
```

`eval/testset.jsonl`，一行一条。注意第 4 条 `expected_doc_ids: []`——这是**故意的「答不出」用例**，用来验证兜底。

---

## 2. 要覆盖哪些类型

20 条要有代表性，按这几类配比：

| 类型 | 占比 | 作用 | 例子 |
|------|------|------|------|
| 直球问题 | ~40% | 基本检索能力 | 「年假几天」 |
| 口语/同义问法 | ~25% | 测 query 改写、混合检索 | 「钱能要回来吗」=退款 |
| 跨文档问题 | ~15% | 测多片段召回 | 答案散在两篇文档 |
| 答不出问题 | ~10% | 测兜底（不能编） | 知识库里没有的 |
| 权限相关 | ~10% | 测越权不召回 | 普通员工问薪资细则 |

**口语/同义和答不出这两类最容易被忽略，但恰恰最能暴露问题**——直球问题谁都能答对，区分度低。

---

## 3. 怎么造出这些用例

三个来源，从易到难：

### 来源一：人工根据文档造（起步）

浏览你的 demo 文档，每篇想 1-2 个「员工会怎么问」的问题，记下它应命中哪篇。20 篇文档 → 20-40 条用例。**起步最快。**

### 来源二：让 LLM 辅助生成（提速）

给模型一篇文档，让它生成「员工可能会问的问题」：

```python
# 用 LLM 从文档反向生成问题
prompt = f"""这是一篇企业文档：
{doc_content}

请生成 3 个员工可能会问、且答案能在本文档找到的问题。
再生成 1 个口语化、用词和文档不同的问法。
输出 JSON 列表。"""
```

生成后**人工审一遍**（剔除不合理的），效率比纯手写高。

### 来源三：真实日志（最有价值，上线后）

上线后从 `query_logs` 里捞真实问题，尤其是用户标了「没用」的——这些是系统的真实弱点，补进评估集最值钱。详见 [10-production/03-feedback-loop](../10-production/03-feedback-loop.md)。

## 4. 为什么要标 `asker_role`

因为权限。同一个问题，不同角色的「期望命中文档」可能不同：

```jsonl
{"question": "薪资是怎么核算的？", "asker_role": "hr", "expected_doc_ids": [30]}
{"question": "薪资是怎么核算的？", "asker_role": "engineer", "expected_doc_ids": []}
```

HR 问能命中薪资细则（doc 30），普通工程师问**应该命中不到**（无权限 → 期望为空 → 应走兜底）。这一对用例直接验证了「权限过滤生效」。没有 `asker_role`，权限根本没法评估。详见 [08-permission/05-no-leak-verify](../08-permission/05-no-leak-verify.md)。

## 5. 评估集的质量自查

造完 20 条，问自己：

- [ ] 有没有口语/同义问法？（不能全是直球）
- [ ] 有没有「答不出」用例？（验证兜底）
- [ ] 有没有权限用例？（同问题不同角色）
- [ ] `expected_doc_ids` 是不是真的核对过原文？（标错了评估就废了）
- [ ] 覆盖了主要文档空间（hr/it/product）吗？

**最关键的是第 4 条**：期望文档标错，整个评估就是错的标尺，比没有还糟。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 全是直球问题 | 区分度低，测不出改进 | 加口语/同义/跨文档 |
| 没有答不出用例 | 兜底没法验证 | 留 ~10% 期望为空 |
| 不标 asker_role | 权限无法评估 | 每条标提问角色 |
| expected_doc_ids 没核对 | 标尺本身是错的 | 逐条核对原文 |
| 追求一次造 200 条 | 迟迟开始不了 | 先 20 条，持续补 |
