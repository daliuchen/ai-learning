# EKB 17：评估先行——为什么标尺要在写检索之前立

> **一句话**：大多数人把评估放在最后（「先做完再测」），这是错的。评估应该是**第一段代码**。没有标尺，你后面每一次「优化」都是凭感觉，根本不知道改完是变好还是变坏。先立标尺，后续所有迭代才有意义。

---

## 1. 没有评估，你在闭眼开车

设想没有评估集的迭代过程：

```
改了分块策略 → 跑几个问题 → 「感觉答得还行」→ 留下
加了 rerank   → 又跑几个   → 「好像更好了？」→ 留下
调了 prompt   → 再跑几个   → 「诶这个变差了？还是我记错了」→ ？？？
```

全靠「感觉」和「记忆」，而人的记忆和感觉极不可靠。**你无法回答「这次改动到底让系统变好了多少」这个最基本的问题。**

有了评估集：

```
baseline: recall@5 = 0.62, 引用准确率 = 0.71
加 rerank → recall@5 = 0.78 ✅ 涨了 0.16，留下
调 prompt → 引用准确率 = 0.68 ❌ 跌了，回退
```

每一步都有数字。这就是「工程」和「玄学」的区别——也是 [04 手册](/docs/04-prompt-engineering/01-foundations/05-eval-first) 反复强调的核心。

---

## 2. 为什么是「先」，不是「后」

「先做完功能再补评估」听起来合理，但有三个致命问题：

| 后置评估的问题 | 前置评估的好处 |
|----------------|----------------|
| 做的时候没标尺，全凭感觉堆 | 每个增强都能验证是否真有用 |
| 做完才发现方向错了，返工 | 早期就发现「这条路不通」 |
| 评估集事后补，容易迎合现有实现 | 评估集独立于实现，更客观 |
| 选型（embedding/模型）没依据 | 用评估集做选型决策 |

**评估集是你的「需求规格说明书」的可执行版本**——它定义了「什么叫答得好」。先写它，等于先把目标钉死。

---

## 3. 评估在本项目的多重用途

评估集不只是「最后打个分」，它贯穿整个项目：

- **选 embedding**（03 章）：哪个模型 recall 高
- **选生成模型**（03 章）：哪个模型引用准、不乱编
- **验证每个检索增强**（07 章）：BM25/rerank/改写各涨多少
- **验证权限**（08 章）：越权文档是否真的不召回
- **上线前 gate**（10 章）：指标达标才发布
- **回归测试**：改代码后跑一遍，确保没退化

一份评估集，回报贯穿全程。这就是为什么它值得最先做。

---

## 4. 评估的两个层次

RAG 系统要分两层评估，因为问题可能出在两个地方：

```
检索层评估：召回的片段对不对？
   指标：recall@k（该命中的文档有没有被检索到）
   ↓
生成层评估：基于片段的答案对不对？
   指标：引用准确率、答案正确性、兜底正确率
```

**分层很重要**：如果答案错了，要能区分是「检索没召回对的片段」还是「片段对了但模型答歪了」。两者的修法完全不同——前者改检索，后者改 prompt/模型。详见 [03-metrics](./03-metrics.md)。

---

## 5. 评估不必一开始就完美

有人因为「评估集很难建得全面」而拖延。错了——**先有 20 条，胜过没有**。

```
第一版：20 条最典型的问题 + 期望文档，够你开始量化迭代
后续：从 query_logs 里发现答不好的真实问题，持续补进评估集
```

评估集是**活的**，随项目成长。关键是**现在就有一个能跑的版本**，而不是等一个完美的。详见 [02-build-testset](./02-build-testset.md)。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 评估放最后 | 全程凭感觉，方向错了才发现 | 评估是第一段代码 |
| 只评估答案，不评估检索 | 答错了不知道哪层的锅 | 分检索/生成两层评估 |
| 等评估集完美再开始 | 永远开始不了 | 先 20 条，后续持续补 |
| 评估集迎合现有实现 | 测不出真问题 | 评估集独立定义「什么叫好」 |
