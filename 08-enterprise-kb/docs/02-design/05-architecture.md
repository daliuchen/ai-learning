# EKB 10：整体架构——数据流与组件职责

> **一句话**：把前几篇的需求、约束、数据模型，组装成一张完整的架构图。本篇讲清两条核心数据流（Ingest 离线流、Query 在线流）各自经过哪些组件、每个组件干什么，作为后续所有章节的「地图」。

---

## 1. 全景架构

```
┌─────────────────────────────────────────────────────┐
│  前端 (Next.js / TS)                                   │
│  ├─ 问答页：输入 → SSE 流式答案 → 引用卡片               │
│  └─ 文档后台：上传 / 设可见角色 / 下线                   │
└──────────────────────┬──────────────────────────────┘
                       │ HTTP（提问/管理） · SSE（流式答案）
┌──────────────────────▼──────────────────────────────┐
│  AI 服务 (FastAPI / Python)                            │
│                                                       │
│  ┌─── Ingest 流（离线）────────────────────────────┐   │
│  │ parse → chunk → embed → upsert(chunks, acl)    │   │
│  └────────────────────────────────────────────────┘   │
│                                                       │
│  ┌─── Query 流（在线）─────────────────────────────┐   │
│  │ rewrite → ACL 过滤 → 混合检索 → rerank          │   │
│  │ → 裁剪 → Pydantic AI 生成（答案+引用+found）     │   │
│  └────────────────────────────────────────────────┘   │
└──────────────────────┬──────────────────────────────┘
                       ▼
              Postgres + pgvector
   documents · chunks · acl · eval_cases · query_logs
```

记住一句话：**Ingest 是离线把文档变成可检索片段，Query 是在线把问题变成带引用的答案。** 两条流共享同一个数据库，但触发时机和频率完全不同。

---

## 2. Ingest 流（离线，文档变更时触发）

```
文档（md/html）
  │ parse        解析成纯文本 + 结构（标题层级）
  ▼
原始文本 + 结构
  │ chunk        按语义/小节切片，每片带 section_path
  ▼
chunks（带元数据）
  │ embed        每片生成向量
  ▼
向量 + 文本
  │ upsert       写入 chunks 表（含 tsv），写 acl 表
  ▼
Postgres
```

特点：**不在用户请求路径上**，可以慢、可以批量。文档更新时增量重跑（详见 [05-ingest/06-incremental](../05-ingest/06-incremental.md)）。

---

## 3. Query 流（在线，每次提问触发）

```
问题 + 用户角色
  │ rewrite       口语 → 接近文档表述的 query（07章）
  ▼
改写后的 query
  │ ACL 过滤      只在该角色可见的 doc 范围内检索（08章）
  ▼
候选 doc 集合
  │ 混合检索       BM25 + 向量，各召回 N 条（07章）
  ▼
召回的 chunks（含噪声）
  │ rerank        精排，选 top-k（07章）
  ▼
top-k chunks
  │ 裁剪          按 token 预算裁，拼进 prompt（07章/07手册）
  ▼
prompt
  │ generate      Pydantic AI → {答案, 引用, found}（06章）
  ▼
结构化答案 → SSE 流式返回前端 + 写 query_logs
```

特点：**在用户请求路径上**，延迟敏感。每一步都要算时间预算（详见 [10-production/02-cost-latency](../10-production/02-cost-latency.md)）。

---

## 4. 组件职责清单

| 组件 | 职责 | 章节 |
|------|------|------|
| parse | 文档 → 纯文本 + 结构 | 05 |
| chunk | 文本 → 带元数据的片段 | 05 |
| embed | 片段 → 向量（可切换模型） | 03/05 |
| rewrite | 口语问题 → 检索友好 query | 07 |
| ACL filter | 按角色限定可检索范围 | 08 |
| hybrid retrieve | BM25 + 向量两路召回 | 07 |
| rerank | 召回结果精排 | 07 |
| generate | 拼 prompt → 结构化答案 | 06 |
| eval | 离线给检索/答案打分 | 04 |

每个组件都是**可独立测试、可单独替换**的——这是「自写薄层」相比「LangChain 黑盒」的最大好处。

---

## 5. 为什么这样分层

| 设计 | 好处 |
|------|------|
| Ingest / Query 两流分离 | 离线慢任务不拖累在线延迟 |
| 检索内核独立于生成 | 检索可单独评估、单独优化 |
| ACL 作为独立过滤步 | 权限逻辑集中，不散落在各处 |
| 生成层用结构化输出 | 引用/兜底是类型保证，不靠 prompt 祈祷 |
| 共享一个 Postgres | ACL 能和检索 join，少一个组件 |

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| Ingest 和 Query 混在一条流 | 文档处理拖慢问答 | 离线/在线分离 |
| ACL 过滤放在检索之后 | 越权文档进了候选集才被滤，有泄漏风险 | 过滤前置到检索范围 |
| 检索和生成耦合 | 无法单独评估检索质量 | 分层，检索可独立打分 |
| 没有统一架构图 | 各章代码拼不起来 | 用本图当地图 |
