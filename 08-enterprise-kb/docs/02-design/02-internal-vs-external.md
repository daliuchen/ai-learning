# EKB 07：对内知识库 vs 对外客服——为什么差别巨大

> **一句话**：「给文档做个问答」听起来是一件事，其实分两种产品——**对内员工知识库**和**对外客户客服**。它们的技术重心几乎不重叠：对内的魔鬼在「权限」，对外的魔鬼在「人机交接」。本手册聚焦对内，本篇说清为什么不能混着做。

---

## 1. 一张对比表

| 维度 | 对内知识库（本手册） | 对外客服系统 |
|------|---------------------|--------------|
| 谁在问 | 内部员工（已登录、有角色） | 外部访客（匿名或弱身份） |
| 核心诉求 | 快速查到**自己有权看**的文档 | 解决问题，必要时找到人 |
| 最大难点 | **权限**：谁能看哪些文档 | **转人工**：什么时候交给坐席、怎么交还 |
| 转人工 | 基本不需要 | 核心功能 |
| 坐席工作台 | 不需要 | 必需（收件箱、会话指派） |
| 会话状态机 | 简单（一问一答） | 复杂（机器人↔人工来回切换） |
| 语气 | 中性、专业 | 友好、品牌化 |
| 答错代价 | 员工操作出错 | 客户流失、投诉、品牌受损 |

**结论**：虽然底层都用 RAG，但「对内」把 80% 精力花在**检索质量 + 权限**，「对外」把一大半精力花在**坐席协作 + 状态机**。

---

## 2. 为什么不建议「一套通吃」

很多人想：做一套知识库，既给员工内部查，又接对外客服，岂不省事？

现实是**两头都做不深**：

- 对内要的**细粒度权限**（按角色/部门过滤文档），对外几乎用不上
- 对外要的**转人工 + Copilot + 坐席台**，对内是纯负担
- 两者的**会话模型**根本不同：对内是无状态问答，对外是有状态的多方会话

硬塞在一起，架构会被两套需求互相拉扯，最后是个四不像。**先把一种做透，另一种需要时再单开。**

---

## 3. 它们唯一真正共享的：检索内核

虽然产品形态差很远，但有一块是**通用、可复用**的：

```
文档 → 分块 → embedding → 向量库 → 混合检索 → rerank
```

这条「把文档变成可检索片段」的内核，对内对外完全一样。区别只在内核**上面**那层：

```
          ┌─────────────────────┐
对内 →     │  权限过滤 + 直接问答    │
          ├─────────────────────┤
          │   共享检索内核         │  ← 本手册 05-07 章
          ├─────────────────────┤
对外 →     │  转人工 + 坐席协作      │
          └─────────────────────┘
```

**好消息**：你跟着本手册把检索内核做扎实了，将来要做对外客服，内核可以直接搬，只需在上面接交接逻辑。

---

## 4. 我们为什么选「对内」当教学载体

| 理由 | 说明 |
|------|------|
| 权限是独特价值点 | 它是「真系统」和「玩具 demo」的分水岭，最值得讲 |
| 主线干净 | 没有转人工/坐席台，RAG 主线不被冲淡 |
| 可量化 | 答得准不准、引用对不对，都能打分 |
| 自包含 | 不需要真上线、不需要外部访客，跟做门槛低 |

对外客服的「人机交接状态机」当然也很有意思，但它会把手册重心从 RAG 拽走，更适合单独成篇。

---

## 5. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 对内系统硬加转人工 | 复杂度暴增，用不上 | 对内聚焦问答 + 权限 |
| 对外系统忽视权限 | 其实对外更要防数据泄漏（但粒度不同） | 对外靠「文档是否对外公开」单一开关即可 |
| 想一套通吃 | 两头都不深 | 先做透一种 |
| 检索内核也分两套 | 重复造轮子 | 内核共享，只分上层 |
