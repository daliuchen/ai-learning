# EKB 08：边界与非目标——刻意不做什么

> **一句话**：一个项目能不能讲透，取决于**敢不敢做减法**。本篇明确列出我们刻意**不做**的东西，以及每个「不做」背后的理由。划清边界，才能把有限的篇幅砸在真正的难点上。

---

## 1. 为什么要先定「非目标」

需求会无限膨胀。「既然做了知识库，要不要再加个……」这种想法一旦放任，项目永远做不完，手册也永远讲不清。

**先定非目标，等于给项目装了护栏**：任何新需求来了，先对照非目标列表——在表里的，直接拒绝；不在的，才评估。

---

## 2. 我们不做的，以及为什么

| 非目标 | 为什么不做 |
|--------|-----------|
| 对外客服 / 转人工 / 坐席台 | 重心是人机交接，会冲淡 RAG 主线（见上一篇） |
| 多轮对话记忆 | 知识库以单轮问答为主；多轮记忆是 07 手册的主题，这里不重复 |
| Agent 自主调工具 | 「查订单」这类 agentic 能力属于另一个场景；本项目是纯检索问答 |
| 文档自动生成/写作 | 知识库是「读」文档，不是「写」文档 |
| 生产级高可用/扩缩容 | 教学项目，目标是讲清决策，不是扛百万 QPS |
| 多语言 | 聚焦中文场景；多语言是 embedding 选型的一个变量，点到为止 |
| 花哨的前端 | 前端只做到「能用、能演示流式 + 引用」，不做设计稿级 UI |

---

## 3. 但有几件事，看着像「可砍」其实不能砍

做减法不等于乱砍。下面这些是**核心,绝不能省**：

| 看着可砍 | 为什么必须做 |
|----------|-------------|
| 评估测试集 | 没它就是闭眼改代码，是整个项目的标尺 |
| 权限过滤 | 它是对内知识库的核心价值，砍了就是玩具 |
| 引用溯源 | 没引用的答案没人敢信 |
| 答不出兜底 | 编答案在企业场景是事故 |

**减法砍的是「锦上添花」，不是「地基」。** 分清这两者，是项目设计的关键判断力。

---

## 4. 边界的弹性：留好「将来能接」的口子

「不做」不等于「堵死」。好的边界设计，是**现在不做，但不挡路**：

- 不做对外客服，但**检索内核独立**，将来要接坐席层，内核能直接复用
- 不做多轮记忆，但 API 设计**预留 `session_id`**，将来要加历史不用改数据模型
- 不做 Agent 工具，但生成层用 Pydantic AI，**天然支持将来挂工具**

这种「留口子」的设计，比「一步到位做全」更聪明——既不过度工程，又不把自己焊死。

---

## 5. 用一张图固定边界

```
            ┌──────────────────────────────┐
   做  →     │  对内问答 · 权限 · 引用 · 兜底   │  ← 本手册全部精力
            │  评估 · 检索增强 · 增量更新       │
            └──────────────────────────────┘
            ┌──────────────────────────────┐
 留口子 →    │  session_id · 独立检索内核       │  ← 设计上不挡路
            └──────────────────────────────┘
            ┌──────────────────────────────┐
  不做 →     │  转人工 · 坐席台 · Agent 工具     │  ← 明确拒绝
            │  多轮记忆 · 文档生成 · 高可用      │
            └──────────────────────────────┘
```

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 不定非目标 | 需求无限膨胀，做不完 | 先列非目标当护栏 |
| 把地基也砍了 | 系统变玩具 | 区分「锦上添花」和「地基」 |
| 边界堵死没留口子 | 将来扩展要推翻重来 | 不做但不挡路（留 session_id 等） |
| 过度留口子 | 为「将来可能」过度设计 | 只留低成本、明显会用到的口子 |
