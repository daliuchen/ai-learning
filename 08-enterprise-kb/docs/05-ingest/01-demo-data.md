# EKB 21：准备 demo 数据——一份可信的企业文档集

> **一句话**：本项目用**通用企业 demo 数据**（HR/IT/产品三类文档），既好懂又能体现权限边界。本篇给出文档集的设计——分几个空间、每个空间放什么、怎么标可见角色——这份数据是后面所有章节的「燃料」。

---

## 1. 为什么用通用 demo 数据

我们刻意不用某家公司的真实文档，而是造一份通用的：

- **谁都看得懂**：HR 制度、IT 流程、产品手册，跨行业通用
- **权限边界天然**：HR 薪资文档只 HR 可见，这种约束自然成立
- **自包含**：跟做的人不需要真实数据就能复现整个项目
- **可控**：能精确设计「答不出」「越权」等评估用例

代价是要自己造，但造一份小而精的 demo 数据，比纠结真实数据脱敏快得多。

---

## 2. 文档空间设计

按 `space`（空间）把文档分三类，对应不同的可见角色：

| space | 文档示例 | 可见角色（acl） |
|-------|----------|----------------|
| `hr` | 考勤制度、年假政策、报销制度 | `all`（公开制度） |
| `hr-private` | 薪资核算细则、绩效评定标准 | `hr`, `finance` |
| `it` | VPN 配置、设备申领、账号开通 | `all` |
| `product` | 产品功能说明、API 文档 | `all`, `engineer` |
| `product-internal` | 未发布功能规划、内部路线图 | `product`, `engineer` |

设计要点：**同一类内容里，区分「公开」和「受限」**（如 `hr` vs `hr-private`）。这样才能造出「同一个问题，不同角色命中不同文档」的权限评估用例。

---

## 3. 一篇文档长什么样

用 Markdown，带清晰的标题层级（语义分块要用）：

```markdown
---
title: 差旅与报销制度
space: hr
roles: [all]
source_url: https://wiki.internal/hr/travel-reimbursement
updated_at: 2026-03-01
---

# 差旅与报销制度

## 1. 适用范围
本制度适用于全体员工因公出差产生的费用报销。

## 2. 报销标准
### 2.1 交通费
高铁二等座、经济舱机票据实报销。

### 2.2 住宿费
一线城市每晚上限 600 元，其他城市 400 元。

## 3. 报销流程
### 3.1 单次额度
差旅报销单次上限 2000 元。
### 3.2 时限
需在出差结束后 15 个工作日内提交。
```

**frontmatter** 里的 `space`/`roles`/`source_url`/`updated_at` 会被解析进 `documents` 和 `acl` 表。**标题层级**（`#`/`##`/`###`）会被语义分块用来切片并生成 `section_path`。

## 4. 配套设计评估用例

数据和评估用例要**一起设计**，确保覆盖关键场景：

```jsonl
// 直球：命中公开文档
{"question":"差旅报销单次上限多少？","expected_doc_ids":[7],"asker_role":"all"}
// 口语化：测改写/混合检索
{"question":"出差花的钱多久能报？","expected_doc_ids":[7],"asker_role":"all"}
// 权限：HR 能命中私有文档
{"question":"绩效是怎么评定的？","expected_doc_ids":[15],"asker_role":"hr"}
// 越权：普通员工命中不到私有文档
{"question":"绩效是怎么评定的？","expected_doc_ids":[],"asker_role":"engineer"}
// 答不出：知识库里没有
{"question":"公司股票什么时候上市？","expected_doc_ids":[],"asker_role":"all"}
```

最后两条是这份数据的「灵魂」——它们让权限和兜底变得可验证。

---

## 5. 数据规模建议

教学项目不需要海量数据，**够覆盖各场景即可**：

```
~20 篇文档，跨 5 个 space
切完 ~150-300 个 chunk
配 20-30 条评估用例
```

这个规模：足够暴露真实问题（口语错配、权限、兜底），又小到能秒级 ingest、快速迭代。真实项目把文档换成你们自己的，流程完全一样。

---

## 6. 常见坑

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 文档没有标题层级 | 语义分块没法切 | 用规范的 #/## 结构 |
| 不区分公开/受限文档 | 造不出权限用例 | 设计 hr vs hr-private |
| 数据和评估集分开造 | 覆盖不到关键场景 | 一起设计，确保有权限/兜底用例 |
| 一上来造几百篇 | 迭代慢 | ~20 篇够用 |
| frontmatter 缺字段 | acl/引用信息丢失 | 补全 space/roles/url/updated_at |
