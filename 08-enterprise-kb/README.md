# 🏢 企业知识库实战（Enterprise Knowledge Base）

> 从零搭一个**带权限、带引用、可评估**的企业内部知识库 RAG 系统——把前 7 本手册学到的零件，拼成一个能跑通、经得起追问的真实项目。

前 7 本手册教你**每个零件是什么、怎么用**：Embedding 怎么选、上下文怎么管、prompt 怎么写、Agent 怎么搭。但「学完零件」和「交付一个系统」之间，隔着三样教程从不讲的东西：**怎么选型、怎么把零件粘起来、生产里会炸在哪**。

这本手册用一条主线把它们补上——围绕**同一个企业知识库项目**，从需求设计 → 选型 → 评估先行 → Ingest → 基础 RAG → 检索增强 → 权限 → 前端 → 生产化，一路做到上线。每一步都给出 trade-off、可量化的提升、以及「为什么这么选」。

## 这个项目长什么样

```
employee（带角色）
   │ 提问
   ▼
┌──────────────────────────────────────┐
│  前端 (Next/TS)  问答页 + 文档后台       │
└───────────────┬──────────────────────┘
                │ HTTP / SSE 流式
┌───────────────▼──────────────────────┐
│  AI 服务 (Python / FastAPI)            │
│  ① Ingest   解析→分块→embed→入库        │
│  ② Retrieve query改写→混合检索→rerank   │
│  ③ Generate Pydantic AI 结构化+引用     │
│  ④ ACL      检索阶段按角色过滤可见文档    │
│  ⑤ Eval     recall@k + 引用准确率       │
└───────────────┬──────────────────────┘
                ▼
        Postgres + pgvector
```

跑通后：员工在问答页提问 → 系统按其角色只检索**可见文档** → 混合检索 + rerank 选出最相关片段 → 模型生成**带引用、答不出就承认**的回答 → 引用可点回原文。

## 写作约定

与本集合其它手册一致：一句话总结 → 概念 → 最小代码 → 进阶 → 生产建议 → 常见坑（表格）→ 下一步。代码尽量可独立运行，错误示范用 `# ❌ / # ✅` 标注。本手册的代码片段会逐章拼成一个完整项目。

pip install -r requirements.txt
